与 Claude Code 团队的 Cat 和 Thariq 的炉边谈话

2026年7月21日

本月早些时候，我在AI工程师世界博览会上主持了一场炉边谈话，与来自 Anthropic 的 Claude Code 团队的 Cat Wu 和 Thariq Shihipar 进行了交流。我们讨论了 Claude Code、Claude Tag、Fable、编程智能体的安全性、评估、工具设计，以及 Anthropic 自身如何使用这些工具。

完整的会议视频现已在YouTube上发布。以下是经过编辑的对话文字稿，附有额外链接和我自己的加粗高亮。

如果你不想观看视频或通读整个文字稿，这里有几个要点说明：

  • Claude Tag（Claude 新的协作式 Slack 集成）现在为 Claude Code 团队完成了65%的产品工程 PR。
  • Claude Code 会先向 Anthropic 员工发布功能，并且只发布那些在该用户群体中展现出用户留存的功能。
  • 对 Claude Code 的关键更改仍需人工审查，但团队越来越多地依赖自动化代码审查来处理产品的“外围层”。
  • 在系统提示词中添加示例不再是最佳实践，对于像 Fable 5 甚至 Opus 4.8 这样的模型来说。Claude Code 的系统提示词最近减少了80%的大小。
  • 同样，列出“不要做X和不要做Y”的清单可能会降低最新模型的输出质量。
  • Dogfooding（内部测试）在 Anthropic 内部被称为“ant fooding”。
  • Anthropic 非常相信他们的auto mode，并将其视为 Claude Tag 的赋能技术。
  • Thariq 建议通过“更有抱负地”选择你承担的工作，来抵消编程智能体引发的Deep Blue效应。
  • Fable 擅长编辑视频，Thariq 曾用它编辑其自身的发布视频。
  • Anthropic（在内部）公开工作的文化是他们成功的关键，这体现在他们在公共 Slack 频道中使用 Claude Tag 的方式上。

过去一年里，你的日常工作发生了怎样的变化？

1:05

Simon： Claude Code 是去年二月推出的——它还不到一年半的历史，最初只是 Claude Sonnet 3.7 发布时的一个要点。既然我们现在有了这些能真正为我们工作的编程智能体，过去一年里你的日常工作发生了怎样的变化？

Cat： 我记得我们刚推出 Claude Code 和 Sonnet 3.7 时，你给它一个任务，就必须密切监视它尝试做的每一件小事。我会极其仔细地阅读每一个权限提示。我经常说“不”——不行，不行，不行，你检查过这个文件了吗？你检查过那个文件了吗？而现在，每一代模型的进步都令人难以置信。我觉得我们都有机会退后一步，把更多繁琐的实现工作委托给 Claude。这为我们腾出了大量时间，去思考更有创造性的工作，比如：既然我们知道 Claude Code 能实现很多功能，那么我们应该为用户提供怎样的正确体验？现在有了 Fable，这又是一个完全不同的阶跃式改进。我们看到，对于许多使用场景，你现在确实可以用 Fable 一次性完成大量功能。

Thariq： 我记得我收到的第一条关于 Claude Code 的短信。我的一个好朋友说：“你得去试试 Claude Code。”那大约是在 Opus 4 发布的时候，我试了试，然后心想：“天啊，我现在得去 Anthropic 工作了。”那就是 Opus 4——很棒的模型，但你当时还在阅读权限提示。我们的遗忘速度真是有点疯狂，我现在觉得，哦，自动模式不是一直都在吗，对吧？我甚至不记得自己按过“是”和“允许”。对我来说，我努力推动自己去做的关键一点是，我们必须做出比以往任何时候都更高质量的工作。输出的质量高得惊人。我一直在用它来大量编辑视频，我会想，好吧，它必须在几个小时内满足我们品牌团队非常严苛的要求，否则我们就做不了。这就是我如何尝试随着 Fable 进行转变的：做出我们有史以来最好的工作，而且速度比以往任何时候都快。

哪些传统的软件工程观念不再适用了？

3:39

Simon： 一年前还成立的软件工程惯例，有哪些在如今的新世界里你认为已经不再适用了？

Cat： 我们看到工程技能集最大的转变之一是：两年前，产品经理通常的做法是花六个月时间与大量客户沟通，与跨职能团队对齐需求，撰写详尽的PRD，在写第一行代码前就制定出精确的实现规格。现在情况完全反过来了。对许多工程师来说，我想给大家的建议是：培养更多的商业嗅觉和产品直觉，去思考我们到底应该构建什么，因为从产生想法到构建完成的时间线被大幅压缩——从六个月到一年，缩短到了可能甚至一周。这意味着我们所有人都需要更好的品味，去判断什么值得构建，什么能真正推动我们业务的增长。所以，产品品味和商业嗅觉的价值提升了，而在大多数产品领域，对执行力的强调略有下降。当然，对于基础设施领域，确保每个细节都正确仍然至关重要。

Thariq： 对我来说，是重写代码现在变成了一件好事。

Simon： 以前最糟糕的做法，现在居然可行了！

Thariq： 没错。所有《人月神话》里说的——永远不要重写——我现在是重写的支持者。如果你有良好的测试套件——我认为重写实际上会迫使你确保拥有良好的测试套件——但我觉得人们低估的是，代码库本身就是一份规格说明，而且可能是你拥有的唯一一份，因为没人能完全了解代码库的每一个分支路径。你可以把这个产物提炼出来，或者创建它的其他版本。我们用 Rust 重写了 Bun，效果很好——它现在对我来说已经上线运行了。

Simon： 你们还没在 Bun-in-Rust 上发布 Claude Code 吧？

Thariq： 内部已经用了。

（实际上，Anthropic 似乎已经在6月17日开始向所有用户推送基于 Bun-in-Rust 的 Claude Code 了。）

非工程师们都在用 Claude Tag 做些什么？

6:36

Simon: 最近另一个重大发布是 Claude Tag —— 这东西发布多久了，至少对我们其他人来说，大概一周了吧？据我所知，Anthropic 内部很多非工程师也在用它。非工程师们用 Claude Tag 来做些什么呢？

Cat: Claude Tag 是一个存在于你团队协作工具中的 Claude。我们上周在 Slack 里发布了它。Claude Tag 的不同之处在于，它默认就是多人协作的。一旦你把 Claude Tag 添加到一个 Slack 频道，你就可以参与讨论，你的队友也可以参与，你们可以一起协作处理 PR。另一个重大区别是，它是主动而非被动的。你可以告诉 Claude Tag：“嘿，监控这个频道里的每一个 bug 报告，提交一个修复它的 PR，并 @ 上次修改这部分代码库的工程师”，然后它就会在频道的整个生命周期内自动执行，你无需手动召唤它。第三个重大转变是，我们为此加入了团队记忆功能。如果你在频道里告诉 Claude Tag 你的偏好，它会为未来的每条消息记住这些偏好。如果你总是希望它调试故障，但不希望它调试警告，只需在频道里用自然语言告诉它，它就会为你和团队里的其他人记住这一点。

在内部，我们将 Claude Tag 视为 Claude Code 的演进。 我们认为这是我们内部工作方式的一次巨大转变。Claude Tag 目前处理了我们产品工程团队 65% 的 PR。

Simon: 是整个 Anthropic 公司，还是仅限于 Claude Code 团队？

Cat: 这仅针对我们的产品工程团队——我们内部的 Claude Tag 版本目前处理了我们产品 65% 的 PR。这是一个巨大的转变；这超过了我们一半的 PR。我们观察到人们在 Claude Code 和 Claude Tag 之间分配工作的方式是：Claude Code 仍然是处理最复杂任务的最佳场所，当你需要与智能体进行交互式迭代时。但 Claude Tag 非常适合让它代表你主动工作，这样你就不再需要为所有出现的、与你正在开发的功能相关的 bug 报告手动启动 Claude Code 了。

Thariq： 再说说非编程场景：比如在这次演讲前，我们问 Claude Tag：“嘿，Fable 什么时候发布？”我们想确保能与公告时间对齐。Claude Tag 会搜索我们的 Slack，看看谁说了什么。作为公司的搜索引擎，它真的很有价值。它拥有你产品的所有上下文，所以你可以问它关于指标的问题——通常做决策时，你希望基于指标数据，所以你会把它连接到事件存储。我见过我们的市场团队做这样的事：“嘿，跟我说说这个功能。”他们不是程序员，但 Claude 是——它可以克隆代码库然后说：“这是这个功能，这是它的样子，这是我使用这个功能的录屏。”它能实现各种各样的事情，而我认为我们还在摸索的早期阶段。

Claude Tag 作为团队协作层

10:06

Simon： 我在使用编程智能体时遇到的一个问题是，我知道如何作为个人使用它们，但我不太清楚如何在团队环境中使用它们。听起来 Claude Tag 是你们目前针对这类事情的团队协作层解决方案。

Cat： 没错。而且我们目前很大一部分会话实际上是多人协作的。也许我说：“嘿，我觉得我们应该在 Cowork 里实现这个新功能”，然后我会把 Claude Tag 拉进来做第一版。接着我会告诉 Claude Tag：“分享你最终实现的录屏”，然后我会把设计同事拉进来看看。他们会调整一下，然后交给工程团队完成最后一步并发布到生产环境。这是一种非常流畅的体验。我们仍在努力理清引导同一会话时的社交互动规范，但我们发现人们只是观察别人如何使用它并遵循那些社交规范——对我们来说，将 Claude Tag 整合到团队中一直相当直观。

Thariq： 这对教导他人非常有帮助，也能减少低效操作，因为大家都在看着你一起使用 Claude，这本身也提升了你使用 Claude 的水平。

这让我想起了 Midjourney 如何通过强制在 Discord 频道公开进行提示，来解决教导人们高级图像提示技巧的挑战。

当构建成本变得如此低廉时，你如何决定哪些功能值得构建？

11:41

我自己也觉得很难判断的一点是，现在构建功能的成本已经大幅下降，那么一个功能是否值得立刻发布？

Simon： 你如何应对工程领域最棘手的问题——优先级排序？当构建一个功能变得如此廉价时，你如何决定哪些功能值得构建和发布？

Cat： 这正是难点所在。我们有几种应对方法。其一，我们每天都在内部“吃自己的狗粮”（dogfood）我们的产品。每当我们希望产品能做到某件事却无法实现时，我们不会去寻找其他解决方案，而是改进我们的产品以支持该场景。我们内部有非常浓厚的“狗粮”文化。在将产品分享给全世界之前，我们会先分享给 Anthropic 内部的每一个人，以及一些早期客户，他们会给我们非常诚实的反馈——越犀利越好——然后我们不断迭代，直到人们真正喜欢它。我们内部有一个明确的标准，衡量一个功能需要达到多少活跃用户和多少留存率，才能向全世界发布。因为这个标准非常清晰，每个工程师都知道他们需要达到什么目标。我认为这也提升了我们的打磨水平，因为如果一个功能不够完善，用户就会流失——那我们就不应该发布这个功能。

利用内部用户留存率来决定一个功能是否应该发布，这在我看来非常合理。

你有没有一个让你感到惊喜的功能例子？

12:54

Simon： 你有没有一个让你感到惊喜的功能例子？你们推出后，用户参与度爆表——某个原本不太可能发布的东西，最终变成了真正的产品亮点。

Cat: 我确实有一个。我们团队很多人都喜欢远程控制功能。 远程控制允许你使用移动设备或网页浏览器中的 Claude，连接到你命令行界面中运行的本地 Claude Code 会话。我从来没这个需求，因为我都是直接在手机上启动任务，它会在云端会话中运行，不占用我的本地环境——我想是因为我做的都是很简单的编程任务。这个功能我一开始没完全理解；我当时想，大家直接设置远程开发环境不就行了。但实际上，我们推出远程控制后，很多跟我交流的人都告诉我，他们每晚都做的是：把笔记本电脑插上电源，打开一堆远程控制会话，锁上屏幕，然后躺在沙发上用手机控制 Claude Code。所以这成了我们现在重点推进的一个工作流，虽然我最初没理解——但现在我懂了。

Claude Code 中的每一行生产代码都经过人工审查吗？

14:20

这次大会的一个核心主题是代码审查：人们花了多少精力去审查编程代理为他们编写的代码。我非常想听听 Claude Code 团队对此的看法！

Simon: 代码审查是如何进行的？每一行进入 Claude Code 的生产代码都经过人工审查吗？ 如果不是，你们是怎么做的——如何保证质量？

Thariq: 这很大程度上取决于任务。对于重要领域，我们有代码负责人。 系统提示词就是一个例子，我们有专门的代码负责人——你真的需要获得他们的批准。

Simon: 所以代码负责人直接负责该代码区域的质量。

Thariq: 没错。

Cat: 而且任何涉及该区域的 PR 都需要他们批准。

Thariq：我们使用代码审查 GitHub 机器人来审查所有内容——它会出现在每个 PR 上，而且通常承担了大部分审查工作。我在团队中观察到，对于更复杂的 PR，你可能会制作一个说明文档来解释 PR，以便其他人可以进行审查。我们在验证、CI/CD 等方面投入了大量精力，以确保任何时候出现失败，我们都有相应的测试。我们有一个非常健壮的环境，Claude 可以控制 Claude Code 并对其进行测试。因此，我们采用多管齐下的方法进行代码审查。

Cat：总的来说，我们正努力迈向一个不再需要人类参与其中的未来。对于 Claude Code 核心以及其他产品核心的最关键变更，总会有一个代码所有者，他们会手动审查所有变更。但越来越多地，对于外围层的变更，我们实际上让 Claude 代码审查工具完全负责审查。这听起来可能有点吓人，但我们已经为此进行了长达六个月以上的准备，并且通过循序渐进的步骤来逐步建立对代码审查的信任。最初，所有变更都需要人工审查，然后我们逐渐会说，好的，对于只涉及这些文件的代码变更，代码审查已经能捕捉到 100% 的问题——所以我们实际上不需要人工手动审查这些了。当我们进行事故复盘时，我们会查看导致事故的 PR，然后说，好的，我们该如何更新代码审查来捕捉这类问题？——然后我们会将这些 PR 添加到评估集中，以确保我们未来对代码审查的改进不会在这些指标上出现倒退。将人类从代码审查流程中移除是一个巨大的进步。这听起来可能有些吓人，而且不是一朝一夕就能完成的事情，但通过在基础设施上进行数月的投入，你可以逐步建立起信心，相信代码审查能捕捉到所有你关心的问题。

所以关键似乎在于不断迭代自动化审查系统本身，以便随着时间的推移逐步建立对它们的信任。

新模型如何影响你对其能力边界的直觉判断？

17:20

我们深入探讨了评估（evals）——这是整个大会期间另一个热门话题。

Simon： 我知道 Opus 4.8 如果让我构建一个运行 SQL 查询并输出 JSON 的 JSON 端点，它肯定能直接搞定——我不需要反复检查。但新模型出现时，我就不知道如何快速建立对 Fable 的信任，确保它不会搞砸 Opus 能处理好的事情。新模型如何影响你对它能力边界的直觉判断？

Cat： 我们逐步建立评估基准库，主要是为了让新模型能即插即用。每当有新模型，我们会运行整个评估集，确保例如 Fable 严格优于 Opus 4.8——这给了我们直接替换的信心。

Simon： 这些模型评估是针对整个 Anthropic 的，还是 Claude Code 团队特有的？

Cat： 两者都有。我们团队内部有评估，也会对 Anthropic 内每个代码库进行代码审查评估。对于自动模式这类功能，我们不仅对 Anthropic 内所有用户进行评估，还委托了多个外部测试团队进行红队测试，构建包含提示注入和恶意输入的环境，确保自动模式不会放过任何风险。

如何确认系统提示词的调整确实提升了输出质量？

18:41

Simon： 我想知道对系统提示词的改进是否真的优化了产品——这是最基础的产品级评估，但我仍不确定如何有效实施。你们是否通过某种方式确保对系统提示词的每次调整都能带来更好的输出？

Cat: 我们并非完全有把握，但确实投入大量精力确保性能不会倒退。起点是我们信赖的一套外部评估体系，并辅以更庞大的内部评估体系作为补充。首先，我们主要针对能力进行优化：给定完整的任务定义和全部代码库，Claude能否做出正确决策、彻底修复缺陷并通过所有测试？这是我们优化的起点和核心，因为这直接关系到用户需求。但许多行为会影响用户使用Claude Code时的体验。例如，用户非常不喜欢Claude Code说"该休息了"。或者当它说"嘿，我完成了五个部分中的两个——要继续吗？"时，用户也会感到不满。当然要继续。因此，我们正在建立一套行为评估体系来捕捉这些问题。随着用户反馈的涌入——请务必积极向我们反馈——我们会对优先级问题进行排序，逐一为每个问题建立评估机制。虽然尚未实现100%覆盖，但提升覆盖率是我们的优先事项。

Claude Code团队与模型训练团队之间有多少互动？

20:21

Simon: Claude Code团队与Anthropic内部训练模型的团队之间有多少互动？双方合作紧密吗？

Cat: 在Anthropic内部，我们始终保持紧密协作。经常开会讨论下一代模型预期能实现哪些功能。我们的研究团队在公开分享方面也做得非常出色——博客文章中经常提及我们正致力于攻克日益复杂的长期任务，以及如何训练Claude做到诚实、无害、有益。我们还投入大量精力确保模型能理解用户意图，即使表达模糊。当然，用户最好能明确说明需求，让Claude掌握完整背景——但即便不够明确，我们也会教导Claude做出合理推断。这是一段富有成效的合作关系。

系统提示词精简了80%——你们删减了哪些内容？

21:24

这部分内容包含超多实用提示技巧！

Simon: Thariq，你今早提到，由于Claude Fable的出现，Claude Code的系统提示词减少了80%。能详细说说吗？具体删减了哪些内容？

Thariq: 这不仅归功于Fable——Opus 4.8也起了作用，而且未来的新模型也会延续这个趋势。我们现在为不同模型配置了不同的系统提示词。我们发现的一个规律是，之前对Claude限制过多。早期的模型（比如Opus 4左右的版本）需要大量示例，而移除示例反而效果显著，因为模型的创造力其实比我们提供的示例更丰富。

Simon: 这很有意思，因为我常给人的提示词建议第一条就是：多给示例。如果这个经验不再适用，那我的提示词方法论可能得调整了。
