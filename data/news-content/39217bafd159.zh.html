过去有很多人联系我，询问我的本地代理工具栈以及我是如何搭建它的。

因此，我觉得整理一个简短教程，介绍如何使用开源工具和开放权重的LLM来搭建一个本地（编程）代理，可能会很有用。

本文是一篇关于使用完全本地工具栈搭建生产就绪编程代理的教程。我们将使用一个本地服务的LLM，配合一个本地的编程工具套件，该套件能够读取文件、进行编辑、运行命令并验证更改，如上图所示。

在这里，我们可以将LLM视为提供推理和代码生成能力的引擎。而周围的工具套件则提供了操作环境，使LLM能够在我们本地的项目中执行有意义的编程工作。

为什么选择本地？对于许多编程工作流来说，本地搭建是诸如 Codex 中的 GPT 或 Claude Code 中的 Opus 等专有服务的一个有趣替代方案。本地搭建是透明、可检查的，并且除了硬件和电费成本外，运行是免费的。它也完全在你的控制之下，你可以以任何你喜欢的方式修改编程工具套件。而且，这非常有趣！

顺便说一句，如果你想了解更多关于编程代理工具套件的背景信息，我在这里介绍了编程代理的核心组件（以及出于学习目的从头构建一个编程代理）：

1. 简介

我必须承认，目前我主要还是在 Codex 和 Claude Code 之间交替使用，作为我的日常工具（也为了跟上不断添加的新工具和功能）。此外，计划限制（尤其是 Codex 的）仍然非常慷慨，我至今还不必担心成本问题。

不过，我也使用本地解决方案有一段时间了，用来测试一些东西，并且因为拥有和使用一个完全本地的搭建（相对于专有服务）在某种程度上给我带来了快乐。

无论如何，本地解决方案正变得越来越有吸引力。一方面在于成本。如果你有硬件，运行它们几乎不花什么钱。当然，还有隐私方面的考量。例如，整理和处理我的收据时，我更倾向于让本地模型来处理，而不是把数据发送给 OpenAI 或 Anthropic。

（另外，如果我们记得 Anthropic 最近为了 LLM 研究而限制了其旗舰模型的性能，那么专有服务未来可能会变得更加受限，因此，熟悉开源权重模型作为备选方案可能是个好主意。）

此外，还有很多很多其他的原因和类似用例。

你使用本地 LLM 和编程工具的动机可能包括：

  • 当达到订阅计划限额时，成本可预测且固定，并且不受 API 价格变动的影响。

  • 可复现性；有时模型升级（例如，GPT 5.4 -> GPT 5.5 -> GPT 5.6）能更可靠地解决你的所有查询，这很好。然而，这也可能破坏现有的工作流程。

  • 在经典的飞机飞行场景中，网络慢或没有网络时可以离线使用，或者在没有 Starlink 订阅的情况下，去林间小屋进行编程/写作静修时使用。

可能还有其他一些原因。

因此，在本文中，我们将设置并使用像 Codex 和 Claude Code 这样的流行工具，配合开源权重模型，并研究使用特定模型的工具（如用于 Qwen3.6 的 Qwen-Code）是否会带来额外的好处。（当然，还有很多其他工具，如 OpenCode、Cline、Pi 和 Noumena Code，但我认为大多数人已经对 Codex 或 Claude Code 形成了肌肉记忆，这使得切换到开源权重模型更顺畅一些。）

2. 编程智能体工具概览

大多数编程智能体工具遵循相似的原则，并且具有大致相同的功能和特性。然而，实现细节可能有所不同，并且某些 LLM 通常主要针对特定工具进行了优化。当然，许多开源权重 LLM，例如 GLM 5.2，也可以运行 Claude Code 等。

不过，如果一个 LLM 开发者也开发了一个编程工具，那么可以比较有把握地假设，他们的模型首先针对自己的工具进行了优化（同时也支持其他工具）。

在这里，我将主要使用 Qwen3.6 配合 Qwen-Coder 编程客户端。不过，我也会介绍其他使用本地 LLM 配合其他智能体工具的选项，例如 Claude Code、Codex 以及越来越受欢迎的 Cline，但稍后会详细讨论。

我主要使用 Qwen-Code 来配合 Qwen 模型工作的原因如下：

  1. 它是开源的，就像 Codex (https://github.com/openai/codex) 一样，但不同于 Claude Code；

  2. Qwen 模型已经针对 Qwen-Code 这个工具框架进行了专门优化（更多信息见下文）；

  3. 我可以在同一台机器上并行运行 Codex（使用最新的 GPT 模型）和 Qwen-Code（使用本地 Qwen 模型），而无需在模型之间手动来回切换。

关于上面列表中的第二点，即 Qwen 模型在 Qwen-Code 中表现更好，Nvidia 的论文 Polar: Agentic RL on Any Harness at Scale（2026年5月）中有一个基准测试显示，Qwen3.5-4B 基础模型在所述 Qwen-Code 框架中具有最佳的编码性能（无论是在他们的 Polar-RL 训练之前还是之后），我将其包含在下方。

上表中的基准测试针对的是较旧的 Qwen3.5 模型，我推测最新的 Qwen3.6 模型经过了进一步优化，以在 Qwen-Code 中表现得特别出色。

不过，Pi (https://github.com/earendil-works/pi) 似乎也是一个非常有趣的候选者，我将来需要尝试一下。

顺便说一句，Qwen3.6 35B-A3B 的下载大小约为 22 GB，大约需要 30-40 GB 的内存，并且在配备 M4 的 Mac Mini 和 DGX Spark 上都能运行得相当快。

根据 Cohere 在六月初分享的最新基准测试，它目前是其尺寸级别中最好的本地模型。

如上所示，Qwen3.6 35B-A3B 在这个规模级别中几乎在所有基准测试中都占据主导地位。不过话说回来，Qwen Code 是一个通用的框架，也支持其他类型的模型。例如，我们也可以在 Qwen Code 中接入 North Mini Code 或 Gemma 4。

在架构方面，Qwen3.6 35B-A3B 模型采用了与 Qwen3-Coder 和 Qwen3.5 类似的混合注意力机制。我在《超越标准大语言模型》一文中对此有更详细的阐述。

或者，如果你不想使用 Qwen3.6，Cohere 的 North Mini Code 可能是目前这个规模级别中最有趣、能力最强的替代方案。我也将在下一节的本地 LLM 部署部分介绍这个模型。

3. 本地 LLM 部署

无论我们使用哪种智能体框架（Qwen-Code、Codex 或 Claude Code），我们都必须先设置一个本地 LLM，例如 Qwen3.6 35B-A3B。

有多种选择可以在本地提供模型服务，比如 Ollama、LM Studio、vLLM、SGLang、MLX 等。从我的《从零构建大语言模型》和《从零构建推理模型》项目中你可以知道，我喜欢自己编写这些代码。从头实现一个模型的好处在于，我们能理解整个技术栈，而且可以对其进行修改、进一步训练和微调。

不过在这里，我们只是寻找一个在推理速度和资源需求方面经过极致优化的模型服务框架，因为我们目前不打算进行任何训练或微调。（作为额外步骤，我们可以将自己从头微调的模型转换并导入到这些高效的服务栈中，但这超出了本文的范围。）

在本教程中，我们将使用 Ollama 作为高效的模型服务引擎，因为它在不同操作系统上安装和使用命令行都相对简单（虽然 LM Studio 也添加了非图形界面的 llmster 客户端，但我对它不太熟悉）。

顺便说一句，我与本文提到的任何工具都没有关联，但 Ollama 的一个优点是它还支持可选的云端托管开源权重模型，包括目前最强的开源权重模型 GLM 5.2，该模型太大，无法在消费级硬件上本地运行。（当然，云端模型不是免费的，订阅计划与 ChatGPT 和 Claude 类似；不过，能有这个选项来方便地在“本地”测试最新的顶级开源权重模型还是很不错的。）

总之，设置 Ollama 非常简单，你可以在他们的 下载 页面找到官方的 macOS/Linux/Windows 下载说明。

安装完成后，我建议下载一个模型进行快速测试。例如，在 macOS 上，我们可以使用 Ollama 应用程序通过图形界面直接下载模型：

或者，也可以通过命令行完成，例如：

在 Linux 机器上，请使用非 MLX 版本：

4. 简单的速度性能评估

在决定是否将一个大语言模型用作本地编码智能体之前，进行一次快速的速度和质量评估通常是个好主意。在这里，对于速度评估，我会关注每秒处理的 token 数量。此外，我还会确保这个性能在（非常）长的上下文下保持稳定，因为这正是我们在智能体编码工作流程中通常要处理的情况（与更简单的聊天机器人不同）。

当然，我们也不希望内存开销失控。

你可以运行我的 ollama_speed_memory_bench.py 脚本做个快速检查。简单来说，它会向 Ollama 模型发送不同长度的提示词（从 1k 到 50k 词不等），并默认要求生成最多 8k 个 token。脚本会报告一些基础统计数据：比如从 Ollama 的提示词评估指标中获取预填充速度，从输出 token 计时中获取生成速度，以及从 Ollama 进程（加上可用的 NVIDIA GPU 显存）获取内存使用情况。

例如，在 macOS 上评估 qwen3.6:35b-mlx 时，如果你已从 https://github.com/rasbt/local-coding-agent-evals 下载或克隆了脚本，我们可以运行以下命令（大约需要 5 分钟）：

uv run speed-memory-benchmark/ollama_speed_memory_bench.py --model qwen3.6:35b-mlx

在 Linux 上，我们可以运行：

uv run speed-memory-benchmark/ollama_speed_memory_bench.py --model qwen3.6:35b

请注意，这假设你已按照上一节的说明下载了相应的模型。另外，根据你的系统配置，如果内存少于 30 GB，你可能需要使用更小的模型，比如 gemma4:e2b，它在处理长上下文时最多占用约 8 GB 内存。当然，还有很多更小的模型，但根据我的经验，它们作为本地编程代理的表现相当糟糕。）

请注意，对于模型来说，RSS 内存报告在 macOS 上并不十分准确（尤其是使用 Metal 后端的 mlx 模型变体），我建议在运行时同时留意活动监视器中 Ollama 的内存使用情况。在这种情况下，内存使用量在 20 - 29 GB 之间波动。

无论如何，关键点是：对于 50k 上下文，Qwen3.6 和 North Mini Code 模型最多使用 30 GB 内存，在新款 Mac Mini 上能以约 40 tok/sec 的速度生成输出，在 DGX 上则是 30 tok/sec。

6. 智能体代码库审计

在完成了搭建本地 LLM 的冗长铺垫后，让我们回到正题——编码代理工具。正如本文开头提到的，我们将使用 qwen-code (https://github.com/QwenLM/qwen-code) 工具，因为 Qwen 模型已针对它进行了优化。

如果你熟悉 Claude Code，这基本上是同一回事，只不过是完全开源的。不过，在接下来的章节中，我也会介绍如何将本地的 Qwen3.6 模型连接到 Codex 和 Claude Code。

请注意，编码工具比 LLM 本身功能强大得多。因此我建议大家要更谨慎地对待你运行的内容和位置。例如，在尝试新的（编码）代理时，我倾向于：

  • 首先审查（开源）代理的代码库。

  • 至少在独立的硬件（例如我的 DGX Spark）上运行，或者在我的机器上使用单独的用户账户和/或虚拟环境。

关于审查，我建议关注数据共享/外传、文件权限的默认影响范围，以及对提示注入的基本防御能力。下图试图总结这些要点。

类似的担忧也适用于本地模型服务引擎（例如 Ollama）。然而，编码代理需要更多关注，因为它们可以直接读取你机器上的数据并操作文件。

要进行基本审计，我建议如下：

  1. 克隆仓库：

git clone https://github.com/QwenLM/qwen-code.git

  1. 请一个你之前使用过的可信代理（比如 Codex 中的 GPT 5.5 或 Claude Code 中的 Opus 4.8）用一个聚焦的提示词来审查它。类似如下：

你正在审计 ./qwen-code，以便我决定是否在本地机器上安装或运行该代理。

仅关注已安装代理对本地机器造成的实际风险，以及产生这些风险的代码路径：

  • 安装脚本和包生命周期钩子

  • 代理执行的 shell 命令

  • 运行时的文件读写边界

  • 密钥处理和环境变量继承

  • 仓库文件、项目指令和工具输出如何影响代理

  • MCP、插件、扩展或工具集成

  • 网络调用和遥测数据

  • 安装后的更新机制

  • 终端转义/输出处理

  • 数据外传和数据驻留

忽略安装严格必需的互联网下载，检查当我通过 Ollama 使用本地模型时，已安装的代理是否能将提示词、文件、遥测数据、日志、标识符或元数据发送到远程服务器。忽略云模型配置。

不要仅从项目所有者推断风险。识别具体的端点、SDK、默认提供商、环境变量、配置默认值和控制网络行为的文档，包括任何在外国运营或由第三方公司运营的端点。

不要进行宽泛的风格审查。不要重构。请产出：

  1. 高风险发现（附文件/行号引用）

  2. 中等风险问题

  3. 网络/数据外传发现，包括任何外国、第三方或与中国相关的端点或默认设置

  4. 在审查前应避免运行的命令

  5. 可降低本地机器风险的设置或环境变量

  6. 简短建议：可在沙箱中安全测试、可安全使用，或不要运行

对于每一项，请说明这是否是编码代理的预期行为，或者是否比 Codex 或 Claude Code 固有地风险更高。

以下是主要发现的摘要（因为完整报告可能有点冗长且对本文来说太长）：

  1. 本地执行 Qwen Code 可以通过其 shell 工具在我们的机器上运行 shell 命令，但除非启用 --yolo 等宽松模式，否则会有严格的审批控制。对于一个编码代理来说，这是意料之中的，而这实际上也是它在实践中真正有用的原因。但当然，如果在没有沙盒隔离或包含敏感信息的完整环境中运行，它就会变得有风险。

  2. 数据外传 即使使用本地 Ollama，Qwen Code 也可能向 Alibaba/Aliyun 端点发送使用遥测和元数据，除非禁用使用统计和遥测功能（下文会详细说明）。这比纯本地设置风险更大，因为模型提示词可能留在本地，但会话 ID、工具元数据、模型信息和本地基础 URL 元数据仍可能离开本机。不过，这在各种工具中也很常见（是的，Codex 和 Claude 也会这样做）。

  3. 文件与密钥边界 工作区文件默认可读，而写入操作通常需要审批，并包含一些覆盖保护措施。这是良好且标准的代理实践。

  4. 提示词注入面 仓库说明、工具输出、MCP 工具、扩展和项目配置都可能影响代理的行为。可以通过上述审批关卡来减少提示词注入攻击。这对编码代理来说是正常的，但默认情况下应将不受信任的仓库视为敌对环境，因为它们可能引导代理通过已批准的工具来读取文件、运行命令或发送数据。

关于第 2 点中提到的主要隐私担忧，大部分可以通过自定义 ~/.qwen/settings.json 文件来解决，内容如下：

{
  "privacy": {
    "usageStatisticsEnabled": false
  },
  "general": {
    "enableAutoUpdate": false
  },
  "telemetry": {
    "enabled": false,
    "logPrompts": false,
    "includeSensitiveSpanAttributes": false
  },
  "disableAllHooks": true,
  "mcpServers": {},
  "artifact": {
    "publisher": "local",
    "autoOpen": false
  }
}

"general": { "enableAutoUpdate": false } 这个设置是一种权衡。安全修复将不会自动安装，但我更倾向于明确控制更新时机，而不是让工具在后台自动拉取并应用新代码。

顺便说一下，cline (https://github.com/Cline/Cline)、Codex (https://github.com/openai/codex) 和 Claude Code 都有类似的遥测数据共享默认设置，需要手动禁用。

（请注意，Claude Code 并没有官方开源其代码库，这使得信任它更加困难，而且它似乎确实会向 Anthropic 和 Datadog 发送数据。）

无论如何，总的来说，Qwen-Code 似乎遵循了标准做法，截至本文撰写时，对于编程智能体而言，没有发现什么非标准的特殊问题。

7. Qwen-Code 设置

如果我们接受已报告的发现和风险（就我个人而言，没看到什么危险信号），现在就可以继续安装，并将我们本地的 Qwen3.6-35B-A3B 模型接入 Qwen Code（以及后续章节中的 Codex 和 Claude Code）。

如前所述，我更倾向于在单独的机器上（对我来说是 DGX Spark，但也可以是单独的 Mac 或 Linux 工作站）试验和运行能够读取和编辑本地文件的编程智能体。或者，作为一个实用的折中方案，我也可以在虚拟机中运行它，或者设置一个单独的 macOS 或 Linux 用户账户。

（我听说有些朋友也会为此租用服务器，比如 Linode 或 Heroku，用于折腾。不过，与其为一台还算够用的机器支付每月的托管费用，我可能宁愿买一个相对便宜的 200-500 美元的硬件盒子，甚至是一台旧的退役笔记本电脑，运行本地的 harness，然后如果你在寻找 GPT 或 Claude 的替代方案，可以通过 Ollama 云模型、OpenRouter 等使用云端托管的更强大的开放权重模型。）

总之，让我们来安装 Qwen-Code。列出的选项包括，例如，

以及

npm install -g @qwen-code/qwen-code@latest

然而，运行上述命令的前提是，发布的构建产物与我们刚刚在 GitHub 仓库中审查的代码一致。如果我们格外谨慎/偏执，也可以从 GitHub 仓库自行构建。请注意，这会更手动/更麻烦一些（我建议一次执行一条命令，而不是将整个代码块复制粘贴到终端）：

# 进入开发目录
cd ~/Developer

# 克隆 Qwen Code 的 GitHub 仓库
git clone https://github.com/QwenLM/qwen-code.git

# 进入克隆好的仓库
cd qwen-code

# 安装 JavaScript 依赖
npm install

# 在本地 dist/ 文件夹中构建 CLI 输出
npm run build

# 如果用户级 bin 目录不存在，则创建它
mkdir -p ~/.local/bin

# 创建一个 qwen 包装脚本，用于从当前源码检出运行 CLI。
# 保留 ~/Developer/qwen-code 目录不变，因为此包装脚本会指向它。
cat > ~/.local/bin/qwen <<'SH'
#!/usr/bin/env sh
exec "$HOME/Developer/qwen-code/scripts/cli-entry.js" "$@"
SH

# 使包装脚本可执行。
chmod +x ~/.local/bin/qwen

# 使 qwen 命令在当前 shell 会话中可用。
export PATH="$HOME/.local/bin:$PATH"

# 验证 qwen 命令是否已找到并能打印版本号。
qwen --version

10. Claude Code 设置

当然，我们也可以使用流行的 Claude Code 代理框架来包装我们的本地 LLM。虽然它非常流行且功能强大，但这可能是我在本地设置中最不推荐的选项，因为其代码库是闭源的。这也意味着我们无法方便地检查或禁用 Anthropic 的数据记录行为。

11. Mac <-> DGX

到目前为止，我们讨论的所有设置都假设我们在运行编码工具的同一台机器上运行本地 LLM。

但是，如果我们对编码代理工具产生了一些信任，并希望在我们的主力 Mac 上使用它，而模型本身托管在另一台机器上，例如一台 DGX Spark，该怎么办？

在我看来，最好（或最方便）的设置是从 Mac 到 DGX 建立一个 SSH 隧道。

首先，我建议在 Mac 上退出 Ollama，或者将下面的 11434 端口改为其他端口。

假设我们在 Mac 上退出了 Ollama 应用程序，检查以下命令是否返回空输出，以表明 Ollama 不可用：

curl http://127.0.0.1:11434/v1/models

然后在 Mac 端的终端窗口中运行以下命令：

ssh -N -L 11434:127.0.0.1:11434 rasbt@DGX-Spark

这条命令的作用是以用户 rasbt 的身份建立到 DGX-Spark 的 SSH 连接，你需要将用户名和机器名替换为你自己的。然后，由于 -L 11434:127.0.0.1:11434 这个参数，命令会将 Mac 的本地端口 11434 转发到 DGX 上的 127.0.0.1:11434。请注意，这就是 Ollama 的地址。

运行 ssh -N -L ... 的终端窗口看起来会像是卡住了。这是正常现象。在你使用 Qwen Code、Codex 或 Claude Code 时，请保持这个窗口开启。按 Ctrl-C 可以停止隧道。

所以，当它运行后，在你的 Mac 上使用以下命令来确认 Mac 是否确实能从 DGX 访问 Ollama 模型：

curl http://127.0.0.1:11434/v1/models

如果该命令返回了 DGX 的模型列表，那么你的 Mac 工具就可以像使用本地服务一样使用 DGX 的 Ollama 服务器了。

然后，就像上面一样使用 Qwen Code 和 Codex 即可。

对于通过 ollama launch claude 使用的 Claude，关键在于 Mac 端的 ollama 命令必须能看到隧道端点。如果需要的话：

OLLAMA_HOST=http://127.0.0.1:11434 \
ollama launch claude --model qwen3.6:35b

12. OpenClaw 和 Hermes 呢？

我们重点介绍了 Qwen Code、Codex 和 Claude Code，因为它们与编程智能体工作流最为契合。OpenClaw 和 Hermes 也很有能力，但它们是更广泛的智能体框架。当你希望一个智能体能协调工具、应用、浏览器、终端以及更长时间运行的工作流时，它们更为合适。

对于编程工作，我建议先从 Qwen Code、Codex 或 Claude Code 开始（还有很多其他有趣的编程框架，如 OpenCode、Cline、Pi 和 Noumena Code）。我会将 OpenClaw 和 Hermes 视为超越编程范畴的、有趣的后续选项，而不是这个本地编程智能体设置的首选基础方案。

13. 总结

这是一篇包含大量信息和配置的长文。如果说有几个主要的收获，我认为关键不在于机械的设置流程，而在于本地运行编程智能体时的考量。也就是说，最重要的部分不是安装某个特定工具，而是理解模型服务层、智能体框架、权限模型，以及如何评估该设置是否真能可靠地解决编程任务。

当然，GPT 5.5 和 Opus 4.8 目前比那些能在 Mac 或 DGX Spark 上运行的小型开放权重模型更强大。但新一代 30-35B 参数范围的混合专家模型（如 Qwen3.6、North Mini Code 和 Nemotron 3 Nano）都极其出色，完全能胜任许多任务。而且，它们通过 Pro 订阅服务运行时的 token 处理速度与 GPT 5.5 相同，因此未必会拖慢你的工作流程。

搭建本地智能体时，除了模型本身，主要考虑的是我们想使用哪个框架。普遍看法是，模型通常针对特定框架的优化程度更高（例如，Qwen3.6 在 Qwen Code 中可能比在 Claude Code 中表现更好）。不过，根据这个小型智能体评估来看，情况未必如此（这只是一个非常小的基准测试，所以请谨慎参考）。因此，如果你更习惯使用另一个你已形成肌肉记忆的框架，比如 Codex 和 Claude Code，那么直接把模型放进去试试看，或许也是个不错的选择！

总之，希望这篇文章对你有用，并激发了你对开放权重模型进行一些探索的兴趣。它们的能力与日俱增，而且出于某种难以言喻的原因，在本地运行模型本身就是一种乐趣。
