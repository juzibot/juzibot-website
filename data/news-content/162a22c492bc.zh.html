K3 是一款前沿质量的开源模型，成本却只是零头。更重要的是，它与 Fable 的互补性极佳，这使得通过任务路由来获取最高质量智能成为可能。


🧭 简而言之：我们在约 1,000 个智能体任务上，将 Kimi K3（开源）与 Fable 5（闭源）进行了对比测试，发现：

  1. 通过在 K3 和 Fable 之间进行路由，我们实现了 93% 的准确率。
  2. 在长周期智能体任务中，成本效益比单独使用 Fable 高出约 50 倍，并且在所有用例中成本都持续更低。

我们如何测量

我们对针对不同工作类型的基准测试取平均值，并让 K3 和 Fable 5 在相同的测试框架下运行。总共约 1,030 个任务，都在真实的智能体循环中进行。

类别测试内容任务数
SWE真实代码仓库的错误修复（SWE-bench 风格）460
终端长周期智能体操作：安全、加密、逆向工程、系统管理89
算法LeetCode / AtCoder 风格的问题100
多语言跨六种语言的实现225
法律法律智能体基准（律师评分的任务）120

在深入结果之前，先快速定义一下。Oracle 路由是一种通过让每个模型都运行任务，然后选择最便宜的正确选项来衡量最佳理论性能的方法（即成本/性能的上限）。在实际的路由器中，你无法让任务在多个模型上运行。路由器会预测哪个模型具有最佳的成本和质量权衡，但这终究是一种猜测。

在这项研究中，Oracle 路由显示 K3 被选中执行了 72-96% 的任务。这表明，通过学习日常任务与真正长尾的前沿工作之间的差异，可能实现近乎完美的路由器。但这需要多一个数量级的路由数据，以及真实世界的性能表现才能最终确定。

K3 是一个好模型。

从宏观视角来看，很容易看着两个模型并称这场对决是平局。例如，看 SWE 这个主要的基准测试，K3 得到了 92.4%，Fable 得到了 92.6%。在我们进行基准测试的五类任务中，两个模型的得分往往相差几个百分点，Fable 在其编程语言广度（多语言）方面略微领先。

人们很容易就此打住，说“它们大致相当”。但关键在于，它们在不同任务类型上各有更优的表现。

两个模型胜过一个

如果你深入单个基准测试内部，会发现除了总体准确率数字之外，还有更多细节。以 SWE 为例，两者在总体上打成平手。但如果按问题领域划分 SWE，就能看出每个模型的闪光点。K3 在符号数学和开发工具方面最为出色；Fable 则在 Web 和数据可视化工作上胜出。同样的模式也体现在多语言测试集中，Fable 凭借其广泛性在 Java、Python 和 C++ 上表现优异，而 K3 在 JavaScript 和 Rust 上则与之持平。

对于需要在终端进行长时间、多轮次操作——驱动 Shell 并探测系统——的任务，K3 展现了其真正实力。它完成了一批 Fable 未能攻克的任务：7z 哈希、FEAL 密码分析、泄露的秘密、一个实时漏洞、失控的异步作业。

在 Fireworks 上，K3 的成本最多可降低 50 倍。🫳🎤

虽然在质量上两者在高水平上近乎打平，但在价格上却相差甚远。

那么，这个巨大的价格差距究竟从何而来？答案是 token 定价、提示缓存以及每项任务的投入量。以 SWE 任务为例，K3 的工作量比 Fable 大得多：每个任务大约需要 55 轮交互和 130 万 token，而 Fable 只需要 21 轮和 13 万 token。但在那些耗时较长的终端任务上，情况则相反：Fable 反而会陷入循环，跑出 64 轮交互和 150 万 token（有时甚至直接超时）。

提示缓存是将这种投入转化为 K3 价格优势的主要功臣：即使 K3 读取的 token 数是对方的十倍，但通过缓存命中，SWE 任务的运行成本仍然低于 Fable。这里存在一个权衡。更多的交互轮数通常意味着每次运行需要更长的墙钟时间，也就是运行更慢。如果你需要在两秒内得到答案，这很重要；但如果你是在后台大规模运行智能体，那么成本低得多才是更重要的。

别选模型，做路由。

如果你把每项任务都交给最擅长它的模型处理，你得到的结果不会是介于两个模型之间，而是会超越两者。

按任务路由总是优于任何单一模型运行：

Oracle 路由器选择 K3，处理 72-96% 的任务流量。通过这种方式构建路由器，你最终获得的整体质量优于单独使用任一模型，而成本接近仅使用成本优化模型的水平。

K3 在所有工作类型上都具有成本优势

将质量和成本放在同一张图上。蓝色的 K3 在所有五个任务类别中都位于红色的 Fable 左侧（即更具成本效益的一侧）。准确率互有胜负：Fable 在多语言任务上领先，K3 在终端和法律任务上领先，其余大致持平。

单一模型既浪费又不再是最佳选择

Kimi K3 与 Fable 组合路由，以最优价格释放它们的最佳性能。

单一模型提供商、追求 token 最大化的时代即将结束。任务级数据表明，这些模型是价格迥异的专家。最佳 AI 不再出自单一实验室，而是模型的混合体。

这在实践中意味着：

  1. 开源模型作为默认选择。 像 K3 这样成本低 50 倍的开源模型应成为你的基础方案，因为无论如何，Oracle 都会将大部分流量导向它。
  2. 路由器是你的护城河。 路由器必须针对你的工作负载量身定制，持续学习任务/模型的分配方式，是你保持领先的最佳机会。
