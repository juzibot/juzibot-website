Vibe coding MenuGen

2025年4月27日

我经常坐在餐厅里，翻看菜单，然后感觉……有点无从下手。Pâté（肉酱）到底是什么？Tagine（塔吉锅）又是什么？Cavatappi……这应该是意面吧？Sweetbread（甜面包）听起来好美味（我超爱吃甜食）。有时候情况会变得有点失控。"用陈年凝乳折叠的块茎，最后注入榛果黄油精华。" 好吧……这到底是什么玩意儿？我花了很多时间在网上搜索食物图片，所以当最近参加 vibe coding 黑客松时，我知道这是我终于能动手开发我一直想要、却无处可寻的应用的绝佳机会。它现在终于问世了，我把它叫做……🥁🥁🥁 …… MenuGen：

MenuGen 非常简单。你拍一张菜单的照片，它就会为菜单上的所有菜品生成图片。它把菜单可视化。显然，它不完全是你在那家餐厅会吃到的样子，但它能给你一个基本概念：有些是沙拉，这是一道鱼，这是一道汤，等等。我个人觉得它特别有用，所以在黑客松（我在本地环境让第一个版本跑了起来）之后，我继续 vibe coding 了一段时间，把它部署上线，增加了认证和支付功能，并让它真正可用。所以，它就在这里了，下次你外出就餐时不妨试试 :): menugen.app！

MenuGen 是我第一个端到端 vibe coding 出来的应用。我（一个喜欢捣鼓但几乎没有实际网页开发经验的人）从零开始，一步步做出了一个人们可以注册、付费、获得实际价值的应用，而我也能从中赚取一笔诚实合理的 10% 利润。这感觉相当酷。但除了这个应用的实用性，MenuGen 对我来说还很有趣，因为它是对 vibe coding 应用的一次探索，也是对当今这种开发方式可行性的检验。因此，我没有直接写任何代码；100% 的代码都是由 Cursor+Claude 写的，而我基本上不太了解 MenuGen 在我习惯的传统意义上是如何工作的。所以，既然这个项目“完成了”（至少第一个版本看起来能用），我想写这篇简短的文章来分享我的经历——对于一个非网页开发者来说，如今 vibe coding 一个网页应用究竟是什么样子。

首先，本地版本。在 vibe 编程中，这算是个相当常见的体验：我在本地机器上运行的首个应用原型几乎没花多少时间。我用了 Cursor + Claude 3.7，给了它应用的描述，它很快就写好了所有 React 前端组件，搭出了一个漂亮的网页，有流畅的多彩字体、小 CSS 动画、响应式设计等等，就是没有实际的后端功能。看到新网站这么快就成型，确实很有吸引力。我感觉自己完成了 80%，但（剧透一下...）其实可能只完成了 20%。

OpenAI API. 大概就是从这里开始，麻烦接踵而至。我需要调用 OpenAI API 来识别图片里的菜单项。我得获取 OpenAI API 密钥。还得绕着弯儿在那些关于“项目”和详细权限的菜单里找路。Claude 总是会编造出那些已经废弃的 API、模型名，还有最近都改了的输入输出规范，搞得人很困惑，不过在我来回粘贴文档折腾了一会儿后，它总算解决了问题。单个 API 调用能用了之后，我立刻就遇到了严重的调用频率限制，每 10 分钟只能发几个请求。

Replicate API. 接下来，我需要根据描述来生成图片。我注册了一个新的 Replicate API 密钥，结果很快也遇到了类似的问题。我的请求不生效，因为 LLM 的知识库已经过时了，而且这次连官方文档都有点跟不上趟了，因为 API 最近改了，现在不直接返回 JSON，而是返回一种流式对象，我和 Claude 都看不懂。然后我又碰上了 API 的频率限制，导致应用很难调试。后来我听说，这些都是这些服务为了防范欺诈而采取的常见保护措施，但也让新用户想正经开始用变得很难。我听说 Replicate 正在转向一种新方式，让你提前购买额度，这或许以后会有帮助。

Vercel 部署. 至少在当前阶段，应用在本地运行良好，我对此相当满意。是时候部署基础的第一版了。注册 Vercel，添加项目，进行配置，指向我的 GitHub 仓库，推送到 master 分支，看着新的部署构建开始……然后报错了。日志显示了一些由于未使用变量和其他类似基础问题导致的代码检查错误，但很难理解或调试，因为一切在本地都运行正常，只在 Vercel 构建时出问题。所以我通过推送假的调试提交到 master 分支来强制重新部署，从而调试这些问题。一旦我修复了这些问题，网站仍然拒绝工作。我问了 Claude，问了 ChatGPT，查阅了文档，还谷歌了一番。一小时后，我终于意识到自己犯了个愚蠢的错误——我的 .env.local 文件存储了 OpenAI 和 Replicate 的 API 密钥，但这个文件（正确地！）属于 .gitignore 的一部分，不会被推送到 git，所以你必须手动导航到 Vercel 项目设置，找到正确的地方，手动添加你的环境变量密钥。我相对较快地理解了这个问题，但我能看出一个有抱负的 vibe 码手可能会在这个问题上卡住一阵子。当部署最终成功时，Vercel 欣然提供了一个 URL。这再次让我惊讶，因为我的项目是一个还没准备好公开的私有 git 仓库。我没意识到 Vercel 会把你的 !私有! 的未完成项目仓库就这样自动部署到一个完全公开且容易猜测的 URL 上，哈哈。

Clerk 身份验证. Claude 建议我们使用 Clerk 做身份验证，我就采纳了。注册了 Clerk，配置了项目，拿到了 API 密钥。这时候 Claude 给我编造了大约 1000 行代码，看起来都是些已经废弃的 Clerk API。我不得不来回复制粘贴大量文档，才慢慢把问题解决。接下来，到目前为止，Clerk 一直运行在“开发环境”部署上。要切换到“生产环境”部署，还得费一番周折。Clerk 要求你必须把应用托管在自己拥有的自定义域名上。menugen.vercel.com 是不行的。所以我得买了 menugen.app 这个域名。然后得把域名绑定到我的 Vercel 项目。然后还得修改 DNS 记录。然后还得选一个 OAuth 提供商，比如我选了 Google。但光是配置这个 OAuth 就是一场 配置大冒险。我得启用一个“SSO 连接”。我还得去 Google Cloud Console 创建一个新项目，添加新的 OAuth 凭据。这里还得等一段时间走审批流程。然后我又得在 Vercel、Clerk 和 Google 这些层层嵌套的设置里来回折腾了好一阵，才总算把它正确配置好。我当时真想把项目给放弃了，但第二天早上醒来后，感觉好多了。

Stripe 支付. 接下来我想添加支付功能，让用户可以购买积分。这意味着要搭建另一个网站、注册另一个账号、阅读更多文档、管理更多密钥。我选择 "Next.js" 作为后端，把 "入门指南" 文档中的第一段代码片段复制粘贴到我的应用里……结果报错了。后来我意识到，Stripe 在选择 Next.js 时提供的是 JavaScript 代码，但我的应用是用 TypeScript 构建的，所以每次粘贴代码片段都会让 Cursor 因为 linter 错误而不高兴，不过在我多次让它 "修复错误" 并威胁要换成 ChatGPT 之后，Claude 还是慢慢把问题搞定了。回到 Stripe 仪表板，我们创建一个产品，创建一个价格，找到价格密钥（不是产品密钥！），然后到处复制粘贴这些密钥和 ID。就在这时，我发现 Claude 用了一个很糟糕的方法来匹配成功的 Stripe 支付和用户积分（它试图匹配邮箱地址，但用户在 Stripe 结账时提供的邮箱可能和他们注册 Google 账号用的邮箱不一致，导致用户可能实际拿不到购买的积分）。我指出了这个问题，Claude 立刻道歉，并通过在请求元数据中传递唯一的用户 ID 重写了正确的实现。它感谢我指出问题，并表示以后会正确处理——我知道这不过是画大饼。不过既然快速测试通过了，再点击几下就能把部署从开发环境升级到生产环境，然后重新创建一个产品、重新设置一个价格、重新复制粘贴所有的密钥和 ID，本地和 Vercel 设置里都要改……最后总算搞定了 :)

数据库？工作队列？ 到目前为止，所有处理都是 "即时" 进行的——就是请求发出去，结果马上返回，没有任何缓存、保存或其他操作。所以结果是临时的，如果响应时间太长（比如菜单太长项目太多，或者 API 延迟太高），请求就可能超时失败。刷新页面的话，所有数据都会消失。正确的做法应该是用数据库来注册和跟踪任务状态，客户端只显示最新的就绪状态。我意识到得从应用市场连接一个数据库，比如 Supabase PostgreSQL（虽然 Claude 曾向我推荐 Vercel KV，但我知道它其实已经被弃用了）。然后我们还需要一些队列服务，比如 Upstash 之类的来实际处理任务。这意味着要增加更多服务、更多登录、更多 API 密钥、更多配置、更多文档、更多痛苦。这负担太重了，还是留到以后再说吧。

长话短说。用 vibe coding 方式开发 MenuGen 作为本地演示令人兴奋又好玩，但一旦部署成真实应用就变得相当痛苦。构建现代应用有点像组装宜家的未来产品——到处都是各种服务、文档、API密钥、配置、开发/生产环境部署、团队和安全功能、速率限制、价格层级……与此同时，大语言模型对一切的了解都有些过时，仔细观察会发现它们会犯些微妙但关键的设计错误，有时还会胡编乱造或忽悠你解决方案。但最让我感兴趣的是，我其实没在代码编辑器里花多少时间，大部分时间都花在浏览器里，在各种标签页、设置间切换，配置和拼凑这个庞然大物。所有这些工作和状态甚至都无法被大语言模型访问或操作——照这样下去，我们怎么能在2027年实现社会自动化？

未来展望。作为一个探索，如果你几乎没有Web开发背景，如今用 vibe coding 方式开发应用会是什么体验，我既惊叹（这真的可行，而且比以前容易快捷多了！）又有些对现状的挫败感。当然，部分痛苦在于这些基础设施根本不是为这种使用方式设计的。其目标用户本是生活在LLM时代前的专业Web开发团队，而不是用 vibe coding 方式独自开发应用的原型开发者。以下是一些能让像MenuGen这样的超简单应用更容易创建的解决方案思路：

  • 有些应用开发平台可能会提供"开箱即用"的一站式解决方案。这看起来像是 Vercel Marketplace 的反面——一种有明确立场、具体化、预配置了所有人想要的基础功能（域名、托管、身份验证、支付、数据库、服务器函数）的平台。如果某个服务能让这些功能变得简单且"开箱即用"，那将会非常棒。
  • 所有这些服务都可以变得更加 LLM 友好。你告诉用户的任何内容基本上都会被直接复制粘贴到 LLM 中，所以你不妨直接和 LLM 对话。你的服务可以配备一个 CLI 工具，后端可以用 curl 命令配置，文档可以用 Markdown 编写。这些对于 LLM 来说都是更友好、更符合人体工程学的界面和抽象层。不要和开发者对话，不要让开发者去访问、查看或点击。直接指导并赋能他们的 LLM。
  • 对于我的下一个应用，我考虑使用基础的 HTML/CSS/JS + Python 后端（类似 FastAPI + Fly.io 的风格？），这比"现代网页开发"中无服务器架构的复杂宇宙要简单得多。像 MenuGen（或类似应用）这样的简单应用，在这种模式下可能会容易得多。
  • 最后，MenuGen 根本不应该是一个功能齐全的应用。"应用"本质上只是调用一次 GPT 来 OCR 菜单，然后通过一个循环遍历结果，为每个项目生成图片并优雅地呈现给用户。这听起来就像一个简单的自定义 GPT（按照 OpenAI 早期发布的原始 GPT"应用商店"的术语）。MenuGen 能只是一个提示词吗？LLM 能不返回文本，而是像 Artifacts 那样返回一个简单的网页来展示结果吗？其他很多应用也可以这样吗？我可以把它作为应用发布到商店，并以同样的方式赚取差价吗？

说实话，我很高兴能通过 vibe coding 的方式，把我第一个高度定制化的应用真正做出来，解决了自己长期以来的需求，还能和朋友们分享。感谢上面提到的所有服务，我用它们来构建了这个应用。原则上，如果其他人也喜欢它，它完全有可能被动地赚点钱——这就是 @levelsio 的梦想。说到底，如今用 vibe coding 来开发完整的网页应用还比较混乱，对于任何真正重要的项目来说都不是个好主意。但其中已经展现出巨大的潜力，我认为整个行业只需要一些时间来适应 LLMs 的新世界。我个人非常兴奋地看到，应用开发的门槛将降至接近于零，任何人都能像发 TikTok 一样轻松地构建和发布应用。这类高度定制化的自动化，可能会成为人类创造力的一块全新画布。
