自 OpenAI 发布 o1 模型以来已近两年，该模型普及了基于大语言模型的推理模型这一概念。大约四个月后，DeepSeek-R1 随之问世，并公开了使用可验证奖励的强化学习（RLVR）方法来训练此类推理模型的细节。

1. 推理模型的简要定义

在讨论几乎任何机器学习或 AI 技术或子领域时，有一个教训是，我们通常不应“按字面意思”理解技术术语。例如，机器学习和 AI 中的（人工）神经网络，其工作原理并不像人类大脑这样的生物神经网络那样字面化。

同样，当我们谈论“推理模型”时，也不应期望这些模型真的像我们人类一样进行推理。在 AI 和 LLM 研究的语境下，“推理模型”指的是一个会输出中间推理轨迹的模型，这个轨迹就像一个中间响应，它会逐步解决一个问题或任务。

2. 训练与推理扩展推理模型简要概述

提升（推理）任务表现主要有两种途径：训练扩展与推理扩展。

我们先简要谈谈训练。

2.1 训练推理模型

简而言之，DeepSeek-R1 提出使用带有可验证奖励的强化学习（RLVR）来训练 LLM，将其转变为推理模型。RLVR 是一种为可验证数据领域提供奖励信号（0=incorrect 和 1=correct）的技术。这里的可验证数据领域是数学（我们可以使用像 SymPy 或 WolframAlpha 这样的符号数学检查器来验证结果）和代码（我们可以使用编译器或单元测试，或像 LeetCode 这样的集成平台）来检查正确性。

值得注意的是，推理过程本身并未用于训练或更新模型。尽管他们尝试将这种中间响应信息用于训练，但 DeepSeek-R1 的论文报告称这对模型训练没有帮助，因此最终没有使用。（是否以及如何通过过程奖励模型将中间推理过程纳入训练信号，是一个活跃的研究领域。）

2.2 “顿悟”时刻

总之，如图 7 所示，仅基于输出奖励进行训练，结果证明已经足够让模型学会如何推理问题，这意味着它将学会撰写中间解释、回溯并进行自我纠正。当模型意识到自己犯了错误并进行自我纠正的这些时刻，被称为“顿悟”时刻。

顺便提一下，虽然 DeepSeek-R1 无疑是更受欢迎的论文，也是那篇引发了人们对可验证奖励强化学习和推理模型开发热情的论文，但还有另一篇论文，Kimi K1.5，在同一天（2025年1月22日）发表于 arXiv。此外，RLVR 这个术语早在两个月前的 Tülu 3: Pushing Frontiers in Open Language Model Post-Training 中就已被提出。

​

DeepSeek R1 最终成为更受欢迎论文的一个原因是，它证明了仅通过纯强化学习（RL）就能实现推理行为。

例如，Tülu 3 和 Kimi K1.5 在经过监督微调的模型之上应用了强化学习。DeepSeek-R1 模型也是从 DeepSeek-V3 基础模型的一个 SFT 检查点开始训练的，并且它包含了一个使用纯 RLVR 训练的 DeepSeek-R1-Zero 变体。R1 Zero 是一个比 R1 更弱的模型，但它表明 RLVR 足以教会模型生成和使用推理轨迹。

虽然 R1-Zero 更多是一个概念验证模型，但请注意，完整的 DeepSeek-R1 推理模型训练流程通常是多阶段的，并且如上所述，要稍微复杂一些。

​

顺便一提，如今的大多数大语言模型实际上都是推理模型，这意味着它们都采用了类似于 DeepSeek-R1 的方式，使用某种形式的 RLVR 进行训练。

2.3 推理扩展简述

除了通过训练改善推理行为外，提升模型性能的另一个杠杆是推理计算扩展。简而言之，这意味着我们在模型训练完成后的使用阶段，投入更多的计算资源，以获得更好的答案。

这本身就是一个完整的主题，你可以阅读我的《大语言模型推理模型推理现状》一文，以获得更详细的概述：

我将在下面总结一下最重要的背景信息。

首先，使用 RLVR 训练模型已经隐含地导致了一种推理扩展形式，因为推理模型在推理过程中通常会输出比传统 LLMs 更多的 token，这意味着我们在推理过程中消耗了更多的计算资源。

其次，我们可以通过推理努力级别进一步调整这种输出长度，但稍后会详细讨论这一点。

第三，还有许多其他的推理扩展技术。其中一种流行的方法是自洽性，它通常被实现为一种多数投票的形式，即对模型进行多次查询，最终答案通过多数投票选出。

这种方法既可以应用于传统 LLMs，也可以应用于推理模型。此外，该方法可以按需使用，并且可以与推理训练结合使用。一个很好的例子是 DeepSeekMath-V2，研究人员在推理模型（专用于数学）之上应用了极端的推理扩展，以在具有挑战性的数学奥林匹克类型问题上实现最先进的性能。

3. 思考 token

你可能在之前的“顿悟时刻”图表中见过 <think></think> 标记。我把对应的图也放在下面了，这样你就不用费劲往上翻了。

​

这些 <think> 和 </think> 标签在推理能力方面只是装饰性的。它们并不会让模型进行推理，也并非实现良好推理性能所必需。即使不使用这些分隔符来训练同一个模型，很可能也能达到相似的基准测试性能。

这些 <think> 标签或标记的主要作用，是标明推理过程的开始和结束位置，以便训练流程或用户界面能将其与最终答案分隔开，并可以选择性地对用户隐藏。（像 ChatGPT 或 Codex 这样的界面通常会这样做。）

这里的关键在于，<think> 标记并没有赋予模型“思考”或推理或更好推理的能力。即使不使用这样的 <think> 标记来训练相同的模型，也能达到相似的基准测试性能。

字面字符串 <think> 和 </think> 本身也没有什么特别之处。另一对分隔符也能起到相同的作用。

顺便一提，实现方式通常是在 RLVR 阶段添加一个格式奖励。因此，不仅仅是根据答案正确性来奖励模型，还会为使用 <think> 标记提供额外奖励，这反过来鼓励模型去使用它们。

例如，在 DeepSeek-R1 中，总奖励的计算方式为

R_total = R_accuracy + R_format

其中格式奖励是一个简单的基于规则的检查，鼓励模型将其推理过程放在：

<think>

推理过程

</think> 内。

4. 推理模式的开关

第一代推理模型是专用的推理模型。我的意思是，有一个 DeepSeek-V3 基础模型，以及一个单独的 DeepSeek-R1 推理模型。

无论提示词是什么，R1 通常都会输出非常冗长的回答，使用大量 token，即使对于简单的提示也是如此。它也缺乏一个内置选项来关闭推理模式。

后来的模型，如 Qwen3 和其他模型，尝试了混合方法，使得同一个模型既可以像常规的指令微调模型那样运行，也可以按需切换为推理模型。

注意：一些模型开发者称之为“思考模式”，而另一些则称之为“推理模式”。这两个术语指的是相同的行为。

在 Qwen3 中，这是通过分词器使用 enable_thinking=True 或 enable_thinking=False 来处理的。在底层，设置 enable_thinking=False 本质上是在助手回复的开头添加一个空的 <think></think> 部分，以关闭 Qwen3 的推理（“思考”）模式。

在训练过程中这是如何实现的，以至于模型在推理时支持这种切换，如上图所示？

简而言之，正如 Qwen3 技术报告 中所述，这种开关行为主要是通过监督微调（SFT）引入的，并在其最大的旗舰模型中通过通用强化学习（RL）得到强化。

例如，在初始推理模型通过长链思维 SFT 和推理 RL 训练后，他们增加了一个“思考模式融合”阶段。在这个额外的 SFT 阶段中，模型会同时看到思考和非思考的示例：

  • /think: <think>{reasoning}</think>{answer}

  • /no_think: <think></think>{answer}

思考是默认行为，因此也可以省略 /think。随后的通用 RL 阶段进一步强化了这种模式和格式遵循。

这些 /think 和 /no_think 标志是一个“软”开关。然而，前面提到的 enable_thinking=False 设置（在 False 情况下会强制添加空的 <think></think>）则充当了一个“硬”开关。

换句话说，分词器不会向查询添加 /no_think。它直接在助手回复的开头填充空的 <think></think> 部分。模型只看到生成的 token，并直接继续生成答案。

无论如何，这种开关本质上是 GPT-5.6 等模型中推理努力级别的简化版本，我将在下一节介绍。

5. “推理努力”设置如何工作

在本节中，我想简要概述不同推理努力开关的可能实现方式，这些开关已在 GPT 5 等模型中引入，并且几乎存在于当今任何旗舰模型中。

具体来说，在本文开头，我展示了一张来自 Codex GPT 5.6 界面的图片，该界面允许用户选择多种推理“努力”设置。

接下来的小节将说明这些设置可能如何实现。然后，在下一节中，我将介绍一些与此主题相关的、更有趣的研究论文。

5.1 推理强度与响应长度及质量

遗憾的是，OpenAI 并未公开其强度设置的具体实现细节，但仍有一些证据可供我们进行合理推测。

例如，通过他们去年开源的 gpt-oss 模型（我曾在《从 GPT-2 到 gpt-oss：解析架构演进》一文中讨论过），我们了解到 OpenAI 允许我们通过系统提示词（"Reasoning effort: low/medium/high"）来调整推理强度设置，该提示词会添加在每个用户提示之前。

正如预期，推理强度直接影响响应的长度和准确性，如下所示。

据推测，他们的 GPT 5 模型，包括最近的 GPT 5.6 模型，也采用了类似的方法。

顺便一提，请注意上图中不同的强度设置如何影响响应长度。强度级别似乎与 token 使用量直接相关，而后者又似乎与准确率相关。或许可以设计出比“高”更高的强度设置，但我认为性能在某个点会趋于饱和。这种饱和现象在 GPT 5.6 Sol 模型上更为明显，这也表明增加推理预算在某个点后可能变得不经济。

另一个很好的、非常新的数据点，展示了推理强度、token 使用量和基准性能之间关系的，是本周 Thinking Machine Labs 发布的新开源权重模型 Inkling。

正如本节所讨论的，在推理过程中，推理努力程度可以通过系统提示来简单控制。（ChatGPT 的 UI 可能只是将菜单选项映射到系统提示。）然而，这对于任意模型并不适用，并且需要对训练流程进行某些修改，这将在下文讨论。

5.2 可能的努力程度实现方式

虽然训练细节并未公开，无论是对于 GPT 5.6 还是开源的 gpt-oss 模型，通常，推理努力标签会在后训练期间包含在提示中。

通常有两种实现方式。

第一，我们可以将其作为 RLVR 过程的一部分来实现，并在使用不同系统提示时应用不同的长度惩罚。例如，当提示为“推理努力程度：低”时应用高长度惩罚，而当提示为“推理努力程度：高”时应用轻微或无惩罚。

第二，我们可以在 RLVR 之后通过监督微调对模型进行微调，使其遵循不同的努力指令。

例如，在核心 RLVR 阶段之后，在 SFT 期间，训练数据集中的提示会与展示所需推理量的目标响应配对。（这些目标响应可以由人类撰写，由另一个模型生成，或生成后经过筛选。）

在这个 SFT 阶段，模型直接从训练样本中学习努力标签与目标推理长度之间的关联。而基于 RL 的实现方式则会将努力标签和预算感知的奖励置于 RLVR 阶段内部。这两种方法也可以结合使用，我猜测 gpt-oss 和 GPT 5.6 都采用了这种方式（请注意，GPT 5.6 中的努力设置很可能只是针对特定用户查询更改了系统提示）。

5.3 Inkling 案例研究

刚刚发布的 Inkling 技术报告提供了一个虽小但相当具体的关于努力级别训练的例子。

在大规模 RL 过程中，他们为每个样本做了两件事：

  1. 在系统消息中指定了期望的努力级别。

  2. 调整了分配给每个生成 token 的成本。

从概念上讲，奖励函数可能类似于这样：

\(R(e) = R_{\text{task}} - \lambda(e)N_{\text{tokens}} \)

这里，e 是请求的努力级别，λ(e) 控制 token 惩罚。

  • 低努力使用更高的单 token 成本，以鼓励更短的推理轨迹。

  • 高努力使用更低的单 token 成本，允许模型使用更多的 token。

在推理时，Inkling 会收到类似“思考努力程度：0.8”这样的系统消息，并据此调整其 token 使用量。Inkling 与 gpt-oss、GPT-5.6 等模型的区别在于，其努力程度标签是一个介于 0 和 1 之间的连续数值，而非“低”、“中”、“高”这类序数标签。

这使得 Inkling 的努力程度调节主要处于推理强化学习阶段，而不仅仅是在后续的监督微调阶段。

不过，他们并未透露具体的奖励公式、token 成本系数，也未说明努力程度调节是否也包含在监督微调中。

5.4 关于推理缩放与训练缩放的简要说明

在进入推理努力程度相关论文之前，我想先将本节内容与前面的“2.3 推理缩放简介”部分联系起来。

之前，我将缩放分为训练计算缩放和推理时缩放。GPT-5.6 的界面很好地展示了二者的区别，如下所示。

在左侧，选择 Luna、Terra 或 Sol 会改变模型本身。粗略类比，这对应于训练计算缩放。这些是经过独立训练的模型。在固定的训练方案和数据集大小下，更大的模型需要更多的训练计算。通常，它生成每个 token 所需的计算量也更多。

在右侧，我们保持模型不变，只改变推理努力程度。这就是推理时缩放。模型权重保持不变，但允许模型在回答上花费更少或更多的 token。

一个小小的术语说明是，从菜单中选择不同的模型并非在那一刻进行训练缩放。训练已经完成。更好的理解是，模型菜单是在选择那些在不同训练规模下产生的模型。

下面的 Artificial Analysis 结果展示了这两个轴在实践中如何相互作用。

每条蓝色曲线对应一个模型，Luna、Terra 或 Sol。沿着曲线增加推理努力即为推理缩放。从一条模型曲线移动到另一条，则对应模型缩放，在此我将其作为训练缩放的实用替代指标。

不出所料，两种方法都能提升基准测试分数，但也会增加成本。更有趣的是，这些曲线存在重叠。例如，一个较小模型在较高推理努力下，有时能达到与较大模型在较低推理努力下相似的分数。

顺便一提，此图的 x 轴显示的是 API 成本，而非原始计算量。API 成本是一个实用的衡量指标，但它也取决于提供商的定价和生成的 token 数量。此外，这些曲线的确切形状是特定于基准测试的。

因此，模型大小和推理努力构成了两个独立的调节旋钮。我们可以使用更大的模型、增加推理努力，或者两者结合。哪种组合最佳，取决于所需的精度、成本和延迟。

6.1 DeepSeek V4 训练独立的努力专家

让我们从 DeepSeek V4 技术报告开始，它描述了三种模式的使用：

  • Non-think 直接生成响应，不包含推理轨迹。

  • Think High 是经典方法，模型将推理轨迹置于 <think> 和 </think> 标签之间。这与本文开头 DeepSeek R1 部分（第 2 节）讨论的内容类似。

  • Think Max 与上述相同，但增加了一个特殊的系统指令。（下文会详细说明。）

Think Max 的附加系统提示指令以“Reasoning Effort: Absolute maximum with no shortcuts permitted.”开头。

乍一听，这像是一个简单的提示词工程技巧，但这个提示词背后其实有不同的训练设置。也就是说，每种模式都使用自己的上下文窗口和长度惩罚（遗憾的是，报告中没有详细说明具体的长度惩罚实现）。Think Max 模式比 Think High 模式拥有更长的上下文窗口和更小的长度惩罚，这给了它更多继续推理的空间。

因此，系统指令选择的是在后训练阶段创建的一种行为。将相同的指令添加到任意模型中并不会产生相同的效果。

遗憾的是，这份公开且非常详细的 DeepSeek V4 报告，并未将推理模式和领域专家的描述以足够的细节联系起来，因此无法重建确切的教师分配方案。

然而，报告指出，最终支持不同推理努力水平的模型，是通过从上述教师进行同策略蒸馏而创建的。

总结来说，DeepSeek V4 在后训练阶段开发了三个推理专家模块。从基础模型开始，它先应用监督微调，然后通过 GRPO 进行 RLVR。不同模式的 RL 配置各有差异。具体而言，每个专家模块使用自己的上下文窗口和长度惩罚，而 Think Max 还额外接收一个特殊的系统指令。

接着，包括领域专家在内，不同的推理模式专家模块被蒸馏到一个单一的检查点中，该检查点支持所有三种努力模式。

6.2 Nemotron 3 Ultra 结合学习模式与硬性预算

Nemotron 3 Ultra 技术报告描述了三种设置，分别称为推理关闭、常规和中等努力，类似于上一节中的 DeepSeek V4。与常规模式相比，中等努力是成本更低的推理模式。NVIDIA 在 SFT 阶段引入此模式，使用由 GPT-OSS-120B 在其中等努力模式下生成的示例，然后在 RLVR 期间进一步优化它。大约 2.5% 的 RLVR 提示使用中等努力（这对应于对其奖励应用的基于长度的调整）。

6.2.1 在推理期间使用 Nemotron 推理预算

在推理时，所有三种模式都通过 聊天模板进行选择。

1) 常规模式是默认设置，使用 enable_thinking=True，它会以一个开头的 <think> 标签开始助手的回复。

2) 中等推理强度模式使用 enable_thinking=True 并搭配 medium_effort=True，其中后一个设置还会在最新的用户消息后附加 {reasoning effort: efficient}。

顺便一提，更复杂的是，常规模式和中等推理强度模式还可以与一个独立的推理时间预算相结合。这个预算充当一个外部停止机制。在发布的实现中，聊天客户端会要求模型在接近所选 token 限制时结束推理过程。如果模型尚未输出 </think>，客户端会关闭推理块并继续生成以产生最终答案。学习到的推理强度模式决定了模型如何使用其推理 token，而预算则限制了推理过程可以持续多长时间。这使得可以根据期望的成本和准确性，将任一模式与更紧或更松的预算配对。

3) 关闭推理模式使用 enable_thinking=False，它会预填充一个空的 <think></think> 块（类似于第 4 节讨论的 Qwen3），以便模型直接进入最终响应。因此，这些是聊天模板控制项，而非系统提示。

6.2.2 Nemotron 中的推理预算感知训练

上述推理控制由两个相关的 SFT 组件支持。第一个组件使用 GPT-OSS-120B 的推理轨迹引入中等推理强度行为，如前所述。第二个组件使模型为严格的推理预算做好准备。

为了构建这些训练数据，作者们采用常规推理轨迹，在随机选取的 token 预算处进行截断，并保留原始的最终答案。插入的 </think> token 在 SFT 损失中被屏蔽。因此，模型会看到这样的示例：它必须从一个不完整的推理轨迹，在推理块被外部关闭后，直接跳转到答案。

中等努力训练随后在 RLVR 阶段继续进行。大约 2.5% 的 RL 提示在数学、STEM 和编码任务中使用中等努力设置。报告指出，该模式可以通过奖励超参数进行校准，其中基于长度的奖励调整为成本与质量的权衡提供了额外的控制。

6.3 Kimi K2.5 交替进行有预算和无约束的 RL

Kimi K2.5 技术报告讨论了一种名为 Token Efficient RL 的训练方法，用于降低推理努力。（虽然本周有 K3 的公告，但 K3 的推理努力方法并未公开披露，它可能与 K2.5 相似或相关。）

6.3.1 Kimi 的 Toggle 方法

报告提到，固定的 token 预算可能会使推理模型过度拟合于短解决方案。这意味着模型变得更简洁（即更快、更便宜），但它可能失去从额外的推理时间计算中获益的能力，从而可能导致性能下降。

Kimi K2.5 的方法名为 Toggle，它在每固定数量的训练迭代中交替执行两个 RL 阶段：

1. 在预算阶段，鼓励正确解保持在特定问题的 token 预算内。

2. 在无约束阶段，恢复通常的最大生成长度，以便模型仍能从更长的解中学习。

对于每个问题，其预算根据 RLVR 中正确 rollout 的响应长度所选百分位数来估算。预算约束仅在该问题的平均准确率超过阈值后才激活。这避免了在模型能可靠解决问题之前，就强迫其缩短推理过程。

该报告在 K2 Thinking 上评估了 Toggle，发现它在基准性能变化不大的情况下，将生成的 token 减少了约 25% 到 30%。同样的行为也从数学和编程 RL 任务迁移到了 GPQA 和 MMLU-Pro 上。

Toggle 提供了一个具体的旗舰模型训练方案，用于训练一个 token 使用效率更高、同时保留其在测试时扩展能力的推理策略。

6.3.2 Toggle 在推理阶段带来的变化

Toggle 完全在强化学习训练阶段运行。两个交替阶段更新的是同一个策略（即 LLM），最终的（统一）检查点并不包含“有预算”与“无约束”的选择器。在推理时，生成的模型默认以思考模式运行。

不过有趣的是，Kimi K2.5 本身在我检查的一些 API（如 vLLM 或 SGLang）中，提供了思考模式与即时模式之间的独立二元选择。思考模式默认启用。即时模式则通过 {"type": "disabled"}（在官方 API 中）或 chat_template_kwargs={"thinking": False}（通过 vLLM 或 SGLang 提供模型服务时）来禁用推理轨迹。然而，这些设置与 Toggle 是分开的。

此外，官方 Kimi 报告并未提供针对即时模式的单独训练方案。不过，K2.5 的 SFT 数据是同时使用早期的 K2 模型（直接生成回答，不进行长推理）和 K2 Thinking（生成扩展推理轨迹）生成的。这很可能使统一检查点接触到两种回答格式，类似于上面 Nemotron 3 中的做法。在推理时，聊天模板通过预填充一个开放的 <think> 标签（用于思考模式）或一个空的 <think></think> 块（用于即时模式）来在两者之间进行选择。但遗憾的是，报告并未披露具体的数据混合比例，也未说明是否使用了额外的、针对特定模式的强化学习。

较新的 Kimi K3 提供了更直接的推理时计算量控制接口。 当前的 Kimi Code 文档列出了三个设置：low、high 和 max，其中 max 是默认值。这些通过 reasoning_effort 参数传递。然而，Moonshot 尚未解释这三个计算量级别在训练中是如何创建的。其 发布文章称，这些细节将在未来的 K3 技术报告中公布，因此我会持续关注。

6.4 GLM-5 通过 SFT 引入轮次级与交错思考

GLM-5 技术报告将 GLM-4.5 引入的二元开关式思考开关，扩展到了多轮对话和工具使用场景。它描述了三种相关行为（而非三种努力级别）：

  • 交错思考：在每次响应和工具调用之前插入一个推理块。

  • 保留思考：在此模式下，聊天会跨轮次保留之前的推理块，以便模型稍后可以重用它们。

  • 轮次级思考：这允许在对话中为每个请求单独启用或禁用推理。

在推理时，轮次级思考是实际的开关。在 Z.ai API中，思考默认是启用的，可以通过 thinking: {"type": "disabled"} 为单个请求禁用。托管实现未公开，但开源的 GLM-5 聊天模板展示了使用 Transformers、vLLM 或 SGLang 自托管时的等效机制。

当思考启用时，它以 <|assistant|><think> 开始助手响应；当禁用时，则以 <|assistant|></think> 开始。后者会立即关闭推理块，因此生成过程直接进入最终答案。

报告指出，这些行为是在多任务监督微调阶段与更新后的聊天模板一同引入的。

监督微调之后，GLM-5 会经历推理强化学习、智能体强化学习和通用强化学习。最后的在策略蒸馏步骤使用前几个阶段的检查点作为教师模型。这有助于最终模型恢复在连续强化学习阶段可能减弱的能力。

6.5 Qwen3 使用模式融合与推理时截断

Qwen3 已在第 4 节介绍过，因此这里仅总结与本次比较相关的部分。根据 Qwen3 技术报告，其后训练流程包含四个阶段：长链思维监督微调、推理强化学习、思维模式融合以及通用强化学习。

思维模式融合是实现思考开关的关键阶段。在此阶段，模型通过监督微调，在混合了思考与非思考示例的数据上进行训练。`/think` 示例包含推理轨迹，而 /no_think 示例则以一个空的 <think></think> 块开始，并附带一个简短答案。随后的通用强化学习阶段会强化模型对两种行为的指令遵循和格式遵循能力。

Qwen3 还支持硬性思考预算。当达到设定的阈值时，推理过程会被停止，并在模型继续生成最终答案前插入一个停止思考的指令。报告指出，这种部分推理行为并非显式训练所得，而是在思维模式融合之后自然涌现的。

这使得 Qwen3 拥有了一个习得的开关以及一个推理时预算。它与 DeepSeek V4 和 Nemotron 的方案类似，但更为简单。

6.6 Inkling 基于连续努力值进行强化学习

Inkling 已在第 5.3 节讨论过。简而言之，其 技术报告提到，他们使用连续努力程度调节（值在 0.0 到 1.0 之间），而非固定的努度标签。

在相对较小的初始 SFT 阶段之后，Inkling 的大部分后训练来自异步 RL，包含超过 3000 万次展开。期望的努力程度包含在系统消息中，并且在 RL 期间会根据该值调整 token 长度惩罚。如前所述，更高的 token 成本会鼓励更短的响应。更低的 token 成本则给模型留出更多推理空间。

下表总结了六份技术报告中实际记录的内容。

因此，观察这六个不同的开源权重模型，它们有一个共享的框架。首先，它们通过 SFT 和聊天模板引入努力模式控制。Qwen3 明确混合了思考与非思考示例，而 GLM-5 则添加了交错、保留和轮次级别的思考模式。

第二个共享组件是模式条件化的 RL 阶段，其中上下文窗口和长度惩罚会根据请求的努力程度而变化。DeepSeek V4、Nemotron 3 Ultra 和 Inkling 采用了这种方法。

第三个要素提高了在明确预算下的鲁棒性。Nemotron 在随机截断的轨迹上训练，Qwen3 可以从强制停止的推理片段继续，而 Kimi 则交替进行有预算和无约束的 RL。这些方法有助于在可用推理长度发生变化甚至被缩短时，保持答案质量。

7. 结论

本文中的开放权重示例通过几种不同的机制来实现推理努力。相似的标签背后可能有不同的专家模型、混合的SFT数据、模式条件奖励、硬性token预算，或是这些方法的组合。

很难说哪种方法最好。这些模型在基础检查点、训练数据、后训练计算、基准测试和服务目标上各不相同。它们的报告也省略了许多进行受控比较所需的细节。（此外，可能并不存在一种放之四海而皆准的方法，一种对交互式助手效果很好的方法，可能并不适合长时间运行的编程智能体。）

当然，圣杯是自动化的努力选择。我们之前在GPT 5的Auto模式中见过这一点。这是一个棘手的问题，最终，其实施可能更多是失败而非成功，这就是为什么它被从用户界面中移除了（至少，我再也找不到了）。

在不久的将来，我认为推理努力仍将是一个显式的模型输入，最常见的方式是通过系统提示来传递。然而，围绕大语言模型的智能体封装/框架，或内部路由器，可能会越来越多地从任务状态和可用资源中自动推断出合适的模式和预算（当然，仍然允许用户覆盖）。

我仍然希望努力选择能变得更加自动化。类似于GPT 5的自动模式，一个廉价的模型或路由器可以根据请求、工具状态以及剩余时间或token预算来选择模式，同时仍然允许用户覆盖。如果你想优化延迟或成本，或者追求最大性能，覆盖功能会很有用。

我意识到这是一篇很长的文章，而且可能不是最吸引人的话题。但考虑到关于大语言模型、推理模型和智能体的所有讨论，我认为对推理模型的审视是之前未曾涉及的内容，我希望这是一次独特且有一定用处的概述！
