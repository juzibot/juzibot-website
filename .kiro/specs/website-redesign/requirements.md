# Requirements Document

## Introduction

This feature is a ground-up redesign and frontend rebuild of the existing JuziBot company website (juzibot.com / "句子互动 · JuziBot"). It is explicitly NOT an optimization of the current pages. The objective is to fundamentally rebuild the visual style, information architecture, page layout, and interaction model to international, modern AI-SaaS standards (referencing design languages such as OpenAI, Stripe, Linear, Vercel, Notion AI, and Anthropic), while preserving 100% of the existing content and reusing the existing high-resolution brand and product assets without modification.

The guiding principle is: **preserve content, completely rebuild design; preserve assets, do not preserve the old layout.**

The existing site is a static, multi-page website written primarily in Simplified Chinese (`lang="zh-CN"`). Several sub-pages (under `products/`, `workforce/`, and `enterprise.html`) are generated by a Python build pipeline (`build_pages.py`) from shared snippet functions, and hand-edited root pages (`index.html`, `about.html`, `fde.html`, `industries.html`) coexist with them. All pages share a single stylesheet (`assets/style.css`). The redesign must operate within this static-site and build-pipeline reality and remain deployable through the existing GitHub Actions pipeline.

This document defines the requirements for the redesign. It focuses on WHAT the redesigned site must achieve and preserve, not on the specific visual implementation choices (which are deferred to the design phase).

## Glossary

- **Website**: The complete set of redesigned static HTML pages served from the repository root and its sub-directories (`products/`, `workforce/`, `careers/`, plus root pages).
- **Design_System**: The unified set of visual design tokens and reusable styling rules (colors, typography, spacing, radius, shadows, card styles, button styles) that govern the visual appearance of the Website, expressed primarily through `assets/style.css`.
- **Page_Builder**: The Python build pipeline (`build_pages.py` and its shared snippet functions such as `nav_html`, `footer_html`, `cta_band`, `page_layout`, `feat_grid`, `split_section`) that generates the sub-pages under `products/`, `workforce/`, and `enterprise.html`.
- **Preserved_Asset**: Any original high-resolution media or content fragment that MUST be retained unmodified, including the brand logo (`assets/brand/logo.png`), partner/customer logos (`assets/brand/logos/*`), the official contact QR code (`assets/brand/qr-qiwei.png`), product feature screenshots (`assets/product-shots/*`), product demo chat boxes (`assets/demos/*`), career figure images (`careers/*`), and the inline SVG industry/IM-channel icons.
- **Preserved_Content**: All existing website text copy, including headings, body text, product descriptions, customer evidence statements, metrics, and announcement messages.
- **Navigation_System**: The site-wide top navigation, including the product dropdown, AI-employee (AI 员工) dropdown, top-level links, and the mobile burger menu.
- **Announcement_Bar**: The scrolling marquee notice bar displayed at the top of pages.
- **Contact_Module**: The contact modal that displays the official QR code, together with the trigger controls that open it.
- **Floating_Panel**: The fixed side panel offering 在线咨询 / 预约演示 / 获取方案 actions.
- **Responsive_Layout**: The layout behavior of the Website across desktop (PC), tablet (Pad), and mobile breakpoints.
- **Build_Output**: The set of static HTML files produced after running the Page_Builder.
- **Deployment_Pipeline**: The existing GitHub Actions workflow (`.github/workflows/deploy.yml`) that deploys the `stage` branch and re-runs the Page_Builder when `build_pages.py` changes.

## Requirements

### Requirement 1: Preserve Brand and Product Assets

**User Story:** As the site owner, I want all original brand and product assets reused exactly as-is, so that brand integrity and the high-resolution quality of screenshots and logos are never degraded.

#### Acceptance Criteria

1. THE Website SHALL reference the brand logo, partner/customer logos, official contact QR code, product feature screenshots, product demo chat boxes, and industry/IM-channel icons exclusively from the existing Preserved_Asset files at their current repository paths.
2. THE Website SHALL serve each Preserved_Asset as the byte-for-byte identical source file (verifiable by identical file size and checksum), without re-compression, re-encoding, redrawing, or replacement.
3. WHERE a Preserved_Asset is a raster image, THE Website SHALL render it at a displayed aspect ratio equal to the source file aspect ratio within a tolerance of ±1%, scaling width and height proportionally.
4. IF a redesigned layout requires a different visual presentation of a Preserved_Asset, THEN THE Website SHALL change only the surrounding layout, sizing container, and styling while keeping the source asset file unmodified.
5. THE Website SHALL retain every Preserved_Asset file currently present in the repository, such that the count and paths of Preserved_Asset files after the redesign equal those before the redesign.
6. IF a Preserved_Asset reference cannot be resolved to an existing file, THEN THE Website SHALL be treated as failing this requirement, with the specific unresolved asset path identified.

### Requirement 2: Preserve Website Text Content

**User Story:** As the site owner, I want all existing copy retained, so that the company's messaging, customer evidence, and product details remain complete and accurate after the redesign.

#### Acceptance Criteria

1. THE Website SHALL retain 100% of the Preserved_Content text strings currently published across the existing pages, such that every text string present in the existing site appears on the redesigned site with no string omitted.
2. WHERE the redesign reorganizes copy, THE Website SHALL re-layout each Preserved_Content text string with its character sequence unchanged, neither deleting any string nor modifying its wording, spelling, or punctuation.
3. THE Website SHALL keep every numeric metric, customer evidence statement, and product name character-for-character identical in value and wording to the corresponding item in the existing Preserved_Content.
4. THE Website SHALL present all Preserved_Content in Simplified Chinese with the document language attribute set to the exact value `zh-CN`, consistent with the existing site.
5. IF any Preserved_Content text string from an existing page is absent from the redesigned page during a content-completeness comparison, THEN THE Website SHALL be treated as failing this requirement, with the specific missing string identified in the comparison output.
6. THE Website SHALL preserve the existing `en/` folder legacy English redirect targets so that each previously valid English URL continues to resolve to its current destination.

### Requirement 3: Rebuild the Visual Design System

**User Story:** As a visitor, I want a modern, premium, tech-forward visual style, so that the company is perceived as an international top-tier AI-SaaS provider.

#### Acceptance Criteria

1. THE Design_System SHALL define design tokens in `assets/style.css` covering all six categories: color, typography, spacing, corner radius, shadow, and elevation, with each category containing at least one named token.
2. THE Design_System SHALL define the default theme as a light theme in which the primary page background token has a perceived lightness of at least 90% (near-white) and body content regions maintain a minimum padding/whitespace token of 16px or greater around primary content blocks.
3. WHERE gradients are applied, THE Design_System SHALL restrict each gradient to accent usage such that any single gradient covers no more than 30% of the visible viewport area and is not used as a full-section solid background fill.
4. THE Design_System SHALL limit the accent color palette to a maximum of 3 defined accent color tokens, with all accent usage referencing only those tokens.
5. THE Design_System SHALL define exactly one canonical token set for card styles, button styles, and shadow treatments, such that every card, button, and shadow on every page references only those tokens with no page-local overrides.
6. WHEN any redesigned page is rendered, THE Website SHALL apply only the Design_System tokens defined in `assets/style.css`, such that no element retains styling from the previous visual framework.
7. IF a referenced design token is undefined or fails to load, THEN THE Website SHALL apply the defined fallback token value and render the page without unstyled (browser-default) elements.

### Requirement 4: Rebuild Information Architecture and Page Layout

**User Story:** As a visitor, I want a clear browsing path with a rebuilt page structure, so that I can understand the products and reach a decision efficiently.

#### Acceptance Criteria

1. THE Website SHALL rebuild the HTML structure, module order, and grid layout of each of the 17 redesigned pages (the 7 product pages miaodong, shouhu, canmou, dongxing, miaohui, cli, zhizao; the 6 AI-employee pages sales, marketing, service, government, finance, hr; industries, enterprise, fde, about, and careers index) so that no redesigned page reuses the DOM module ordering of its pre-redesign version.
2. THE Website SHALL provide a reachable navigation path to each of the 18 pages present in the existing site (the 7 product pages, the 6 AI-employee pages, the industries page, the enterprise page, the FDE page, the about page, the careers page, and the home page) such that every listed page is reachable within at most 2 navigation interactions from the home page.
3. WHEN a visitor hovers over or sets keyboard focus on the product navigation entry or the AI-employee navigation entry, THE Navigation_System SHALL display the corresponding dropdown containing one link per sub-page (7 links for the product dropdown, 6 links for the AI-employee dropdown), with each link resolving to its corresponding sub-page.
4. WHEN a visitor activates an internal navigation link, THE Website SHALL load the target page containing the corresponding Preserved_Content and render its primary content within 3 seconds under a 10 Mbps connection.
5. THE Website SHALL preserve every existing internal link target path so that each current internal link resolves to an existing page and no internal link resolves to a missing page (HTTP 404).
6. IF an activated internal navigation link cannot resolve to an existing target page, THEN THE Website SHALL display an error indication informing the visitor that the page is unavailable and SHALL retain navigation access to the remaining reachable pages.

### Requirement 5: Transform Long-Form Content into Structured Presentation

**User Story:** As a visitor, I want dense text turned into scannable structured modules, so that I can absorb information quickly without losing detail.

#### Acceptance Criteria

1. WHERE a block of Preserved_Content is presented as continuous body text exceeding 60 consecutive words, THE Website SHALL re-present that copy using at least one structured format from the defined set: cards, lists, feature grids, timelines, comparison tables, data display modules, tag categories, or FAQ accordions.
2. WHEN content is restructured into a new presentation format, THE Website SHALL retain every sentence, numeric metric, customer evidence statement, and product name from the original Preserved_Content without deletion, summarization, or alteration of meaning.
3. IF restructuring a block of Preserved_Content would require omitting or summarizing any portion of the original copy, THEN THE Website SHALL retain the full original copy and SHALL NOT introduce any text not present in the original Preserved_Content.
4. THE Website SHALL present product, industry, and AI-employee information using shared Design_System structured components, such that each structured component type renders consistently wherever it appears across the Website rather than as per-page one-off markup.

### Requirement 6: Provide Responsive Layout Across Devices

**User Story:** As a visitor on any device, I want the site to adapt to my screen, so that I can read and navigate comfortably on desktop, tablet, or mobile.

#### Acceptance Criteria

1. THE Responsive_Layout SHALL render all page content at every viewport width from 320px to 1920px without clipping, truncation, or text overlap, where the supported ranges are desktop (viewport width 1024px and above), tablet (viewport width 768px to 1023px), and mobile (viewport width 320px to 767px).
2. WHILE the viewport width is at a mobile breakpoint (320px to 767px), THE Navigation_System SHALL present navigation through the mobile burger menu and SHALL hide the full-width desktop navigation links.
3. WHILE the viewport width is at a desktop breakpoint (1024px and above), THE Navigation_System SHALL present the full top navigation links and SHALL hide the mobile burger menu.
4. WHEN the viewport width changes across any defined breakpoint boundary (767px/768px or 1023px/1024px), THE Responsive_Layout SHALL reflow content so that no page body element extends beyond the viewport width and no horizontal scrollbar appears on the page body at any width from 320px to 1920px.
5. THE Responsive_Layout SHALL render every Preserved_Asset contained within the inner bounds of its layout container at all viewport widths from 320px to 1920px, preserving the source aspect ratio so the asset is neither stretched nor compressed.

### Requirement 7: Preserve Interactive Elements

**User Story:** As a visitor, I want the site's interactive features to keep working after the redesign, so that I can still get information and contact the company.

#### Acceptance Criteria

1. THE Announcement_Bar SHALL display the existing announcement messages as a horizontally scrolling marquee that loops continuously without truncating any message.
2. WHEN a visitor activates a contact trigger, THE Contact_Module SHALL open as an overlay and display the official contact QR code (`assets/brand/qr-qiwei.png`) from the Preserved_Asset set at its original aspect ratio.
3. WHEN a visitor activates the modal close control or clicks the overlay area outside the modal content, THE Contact_Module SHALL close and restore the visitor's prior scroll position.
4. WHILE the page is scrolled beyond 200px of vertical offset, THE Floating_Panel SHALL be visible and offer the 在线咨询, 预约演示, and 获取方案 actions.
5. WHILE the page vertical scroll offset is 200px or less, THE Floating_Panel SHALL be hidden and SHALL NOT capture pointer interaction.
6. WHEN a visitor activates a Navigation_System dropdown trigger, THE Navigation_System SHALL reveal the corresponding dropdown menu listing its linked sub-pages.
7. WHEN a visitor activates the mobile burger menu control, THE Navigation_System SHALL toggle the visibility of the navigation links.
8. THE Website SHALL render the interactive product workflow mockup on the 句子秒懂 product page using the existing demo content, scaling it to fit within its container without horizontal overflow at all supported viewport widths.

### Requirement 8: Maintain the Page Generation Build Pipeline

**User Story:** As a maintainer, I want the generated sub-pages to continue being produced by the build pipeline, so that the redesigned site stays maintainable and consistent with the deployment process.

#### Acceptance Criteria

1. THE Page_Builder SHALL generate the full configured set of sub-pages under `products/`, `workforce/`, and `enterprise.html` to their current target paths, with no configured page omitted.
2. WHEN the Page_Builder is executed, THE Build_Output SHALL apply the rebuilt Design_System and rebuilt layout to every generated page, such that each generated page references the rebuilt stylesheet and shared layout and none references a superseded version.
3. THE Page_Builder SHALL produce generated pages whose shared structural regions (navigation, footer, contact module, floating panel) contain the same links, labels, and element structure as those of the hand-edited root pages, differing only in relative path prefixes appropriate to each page's directory depth.
4. WHEN the Page_Builder is executed, THE Build_Output SHALL retain all Preserved_Content for each generated page, with no item dropped, truncated, or altered.
5. IF the redesign introduces a new shared component, THEN THE Page_Builder SHALL expose that component through a single shared snippet function reused across generated pages, with no per-page duplicated markup for that component.
6. IF the Page_Builder fails to write a generated page or to resolve a Preserved_Content lookup, THEN THE Page_Builder SHALL halt and report the failing page and the failure reason, leaving already-written pages intact.

### Requirement 9: Maintain Deployment Compatibility

**User Story:** As a maintainer, I want the redesigned site to deploy through the existing pipeline, so that no deployment infrastructure changes are required to ship the redesign.

#### Acceptance Criteria

1. THE Website SHALL remain a static multi-page site served as files from the repository file structure, requiring no changes to the existing Deployment_Pipeline configuration.
2. THE Website SHALL function without a server-side runtime, using only static HTML, CSS, and client-side JavaScript, such that every page renders correctly when its file is opened directly from the served file structure.
3. WHEN the Deployment_Pipeline deploys the `stage` branch, THE Deployment_Pipeline SHALL pull the `stage` branch, re-run `build_pages.py` if `build_pages.py` changed, and publish a served Website at `/opt/www/juzibot` in which every redesigned page is reachable and returns its rendered content.
4. WHEN a request targets a legacy URL corresponding to an existing redirect stub under `zh/` or `en/`, THE Website SHALL redirect the request to the home page.
5. THE Website SHALL retain the existing redirect stubs under `zh/` and `en/` and the existing `robots.txt` and `sitemap.xml` files at the repository root.
6. IF the Deployment_Pipeline fails at any step while deploying the `stage` branch, THEN THE Deployment_Pipeline SHALL leave the previously served Website at `/opt/www/juzibot` unchanged and surface a failure indication in the pipeline run.

### Requirement 10: Apply Component-Based Frontend Standards

**User Story:** As a maintainer, I want the frontend rebuilt to modern component-based standards, so that the site is consistent, reusable, and easy to extend.

#### Acceptance Criteria

1. THE Design_System SHALL define reusable component styles such that each component type resolves to the same Design_System class/token set wherever it appears across the Website, rendering consistently.
2. THE Website SHALL apply document metadata on every page consisting of exactly one page title (1 to 60 characters) and one meta description (1 to 160 characters).
3. THE Website SHALL assign a page title and meta description that are unique per page across the Website, so that no two pages share an identical title or identical description.
4. THE Website SHALL structure each page using semantic HTML landmarks, including a single `main` region, a `header`, a `nav`, and a `footer`, with sectioning elements introduced by accessible headings.
5. WHERE a visual pattern appears on more than one page, THE Website SHALL implement that pattern with a single shared Design_System rule referenced by every occurrence rather than duplicated per-page styling.

### Requirement 11: Provide Accessible Markup

**User Story:** As a visitor using assistive technology, I want accessible markup, so that I can perceive and operate the site.

#### Acceptance Criteria

1. WHERE a Preserved_Asset image conveys information or function, THE Website SHALL provide a non-empty text alternative that describes the image's meaning or purpose.
2. WHERE a Preserved_Asset image is purely decorative, THE Website SHALL expose it with an empty text alternative so that assistive technology ignores it.
3. THE Website SHALL provide a programmatically determinable, non-empty accessible name for the navigation burger menu, each contact trigger, and the modal close control.
4. WHEN a visitor presses the Escape key while the contact modal is open, THE Website SHALL close the contact modal.
5. THE Design_System SHALL apply text and background color combinations for body copy that meet a minimum contrast ratio of 4.5:1 (WCAG 2.1 AA for normal-sized text).
